## 一、需求背景与目标

### 1.1 核心痛点


| 用户痛点    | 描述                                   |
| ------- | ------------------------------------ |
| 保单堆叠冗余  | 手里有社保、商业补充医疗、百万医疗等多份保障，不知新险种是否"重复建设" |
| 决策缺依据   | 经纪人建议"学平险浪费"，用户无法验证，只能凭经验说话          |
| 保障盲区难识别 | 不知道哪些风险区间已覆盖、哪些仍缺保障                  |


### 1.2 功能定位

**保障重叠度透视镜 (Coverage Overlap Lens)**：通过**差集运算**（现有保障 - 待购保障）量化冗余度，提供「买/不买/换个买法」的决策建议。

### 1.3 产品目标

1. **信息抽取**：从用户自然语言中提取「现有保障」与「待购险种」的结构化信息
2. **缺口探测**：调用 RAG + 推理逻辑，计算保障区间重叠度
3. **决策输出**：以结构化表格 + 证据链形式给出结论与建议

## 三、产品设计

### 3.1 用户旅程（三阶段）

```
Phase 1: 渐进式信息确认 (Slot Filling)
  └── 用户："我有社保、商业补充医疗、百万医疗，还需要买学平险吗？经纪人说浪费。"
  └── Agent：识别实体，发现缺「商业补充医疗免赔额」「是否含门诊」
  └── 反问："为了精准判断，请确认：您的商业补充医疗是否含门诊责任？免赔额是 0 吗？"
  └── 用户："含门诊，0 免赔。"
  └── is_complete = true → 进入 Phase 2

Phase 2: 逻辑推理与比对 (Reasoning)
  └── RAG 检索：学平险条款结构、百万医疗免赔额逻辑、社保范围
  └── 计算区间：A(>1万) 百万医疗覆盖；B(<1万) 补充医疗覆盖；C(身故) 学平险独有
  └── 结论：A/B 100% 重叠，仅 C 有差异

Phase 3: 结构化结论输出
  └── 表格：风险场景 | 现有防线 | 学平险作用 | 重叠判定
  └── 专家建议：买/不买/换（如建议换购少儿意外险）
```

        "existing_coverage_list": {
            "type": "list",
            "item_spec": {
                "name": "险种名称（社保/商业补充医疗/百万医疗/重疾险/学平险/意外险/防癌险等）",
                "deductible": "免赔额（医疗险，0或数字）",
                "has_outpatient": "是否含门诊（医疗险）",
                "coverage_scope": "保障范围简述（可选）",
                "death_benefit": "身故保额（意外/寿险类，可选）",
                "critical_illness_amount": "重疾保额（重疾险，可选）",

### 3.5 同类问题抽象：多险种「是否该买」分析场景

> 将保障重叠度从「学平险 vs 社保+百万医疗」单点场景，抽象为**多种医疗/健康类保险**的通用「是否应该买某种保险」分析能力。

#### A. 问题类抽象

**通用问题模式**：


| 模式      | 用户表述示例                   | 抽象结构                                     |
| ------- | ------------------------ | ---------------------------------------- |
| 已有+待购   | "我有社保、百万医疗，还需要买学平险吗？"    | `existing=[...]` + `pending=?` + 问「买/不买」 |
| 待购+他人说法 | "经纪人说学平险浪费，真的吗？"         | `pending=学平险` + 需从上下文推断 existing         |
| 待购是否值得  | "重疾险值得买吗？""防癌险和百万医疗重复吗？" | `pending=?`，existing 可能为空或由上下文补全         |
| 多选一/换购  | "学平险和意外险买哪个？"            | `pending=[学平险, 意外险]`，比较建议                |


**本质**：在「现有保障集合」与「待购险种」之间，计算**边际贡献率**，输出「买/不买/换个买法」。

---

#### B. 医疗/健康类险种分类与关键属性


| 险种类别     | 典型险种               | 主要保障范围       | 关键 Slot（影响重叠判断） | 与谁易重叠          |
| -------- | ------------------ | ------------ | --------------- | -------------- |
| **基础医保** | 社保、城镇居民医保          | 住院基础报销、封顶线内  | 无（默认有）          | —              |
| **小额医疗** | 商业补充医疗、门诊险、学平险医疗部分 | 0-1万住院/门诊    | 免赔额、是否含门诊       | 彼此、学平险         |
| **大额医疗** | 百万医疗、惠民保、防癌医疗险     | >1万住院、特殊门诊   | 免赔额、保额、续保       | 百万医疗 vs 防癌医疗   |
| **收入损失** | 重疾险、防癌险（给付型）       | 确诊即赔，弥补收入/康复 | 保额、轻中症          | 重疾 vs 防癌（部分重叠） |
| **意外保障** | 意外险、学平险意外部分        | 身故/伤残、意外医疗   | 身故保额、意外医疗额      | 学平险 vs 意外险     |
| **专项保障** | 防癌险、防癌医疗险          | 癌症相关         | 与百万医疗/重疾险的边界    | 防癌医疗 vs 百万医疗   |

#### C. 按险种类的 Slot 与反问策略（扩展）

不同「现有保障 + 待购险种」组合，需要追问的**关键变量**不同：


| 待购险种           | 现有含医疗类保障时需追问         | 现有含意外/身故时需追问 |
| -------------- | -------------------- | ------------ |
| **学平险**        | 商业补充医疗免赔额、门诊；百万医疗免赔额 | 意外险身故保额      |
| **意外险**        | —                    | 学平险/其他意外险身故额 |
| **百万医疗**       | 已有百万医疗的免赔额、续保；惠民保范围  | —            |
| **重疾险**        | 已有重疾保额、是否含轻中症        | —            |
| **防癌险/防癌医疗**   | 百万医疗是否含癌症；重疾险癌症责任    | —            |
| **商业补充医疗/门诊险** | 现有小额医疗免赔额、门诊         | —            |


**反问模板**（按组合动态生成）：

- 医疗类：`您的【险种名】免赔额是多少？是否含门诊？`
- 意外/身故：`您现有的意外险身故保额大约多少？`
- 重疾/防癌：`您已有重疾险吗？保额多少？是否含癌症二次赔？`

#### D. 风险场景与保障区间映射（通用）


| 风险场景          | 覆盖该场景的险种类型                  | 重叠判断要点              |
| ------------- | --------------------------- | ------------------- |
| **小病/感冒/门诊**  | 社保 + 商业补充医疗(0免赔含门诊)、门诊险、学平险 | 免赔额 0 + 含门诊 → 已覆盖   |
| **住院（<1万）**   | 社保 + 商业补充医疗、学平险住院责任         | 补充医疗覆盖则学平险住院重叠      |
| **住院（>1万）**   | 百万医疗、惠民保、防癌医疗险              | 百万医疗与防癌医疗在「非癌住院」重叠  |
| **癌症治疗**      | 百万医疗、防癌医疗险、重疾险(给付)          | 百万医疗已含癌症→防癌医疗重叠     |
| **确诊重疾后收入损失** | 重疾险、防癌险(给付型)                | 重疾险已含癌症→防癌险给付重叠     |
| **意外身故/伤残**   | 意外险、学平险、含身故的重疾险             | 身故额低时学平险与意外险重叠      |
| **意外医疗**      | 意外险、学平险、商业补充医疗              | 0免赔补充医疗 + 意外医疗 → 重叠 |

#### E. 扩展后的推理框架

```
输入：existing_coverage_list + pending_insurance
       ↓
1. 险种分类：将 existing / pending 映射到【基础医保|小额医疗|大额医疗|收入损失|意外保障|专项】
       ↓
2. 风险场景枚举：根据 pending 的类型，枚举其覆盖的风险场景（如学平险→门诊+住院+身故）
       ↓
3. 逐场景对冲：对每个场景，检查 existing 中是否有险种已覆盖
       - 已覆盖 → 重叠
       - 部分覆盖 → 部分重叠
       - 未覆盖 → 无重叠（边际贡献高）
       ↓
4. 边际贡献率：无重叠场景占比 → 高/中/低
       ↓
5. 建议输出：买/不买/换（若重叠高但某场景有差异，建议换购更针对性险种）
```

## 四、技术方案

### 4.1 整体架构

```
                    ┌─────────────────────────────────────────────────────────────────┐
                    │                    保障重叠度透视镜模块（新增）                     │
                    │  extract_coverage_slots() → compute_coverage_gap() → generate... │

```

### 4.2 意图识别扩展

**新增意图**：`INTENT_COVERAGE_OVERLAP`

**触发关键词**（规则模式）：

- 重叠、重复、浪费、冗余、重复建设
- 还需要买吗、要不要买、值不值得买、值得买吗
- 已有社保/百万医疗/补充医疗/重疾险 + 问学平险/意外险/防癌险/门诊险等
- 经纪人说、代理人说、别人说 + 浪费/不用买
- 防癌险和百万医疗重复吗、重疾险和防癌险买哪个

**配置**：`config/settings.py` 或 `app/intent.py` 中追加关键词映射。

### 5.2 多知识库组合调用（可行性说明）

**结论：可行。** RAGflow 的 retrieval 接口已支持 `dataset_ids` 为**数组**，一次请求可检索多个知识库，返回合并后的 chunks。

#### A. API 能力确认

根据 RAGflow 官方文档 `POST /api/v1/retrieval`：

- `dataset_ids`：`list[string]`，可传入多个知识库 ID
- 一次请求会从**所有指定知识库**中检索，按相似度合并排序后返回
- **前提**：多个知识库需使用**相同的 embedding 模型**，否则可能报错

#### B. 当前项目代码支持情况

`app/ragflow_client.py` 中已有：

```python
payload["dataset_ids"] = [kb_id] if isinstance(kb_id, str) else list(kb_id)
```

即：若传入 `knowledge_base_id` 为列表如 `["id1", "id2", "id3"]`，会直接作为 `dataset_ids` 传给 RAGflow，**无需改代码即可支持多库**。

#### C. 多库组合策略建议


| 策略          | 说明                                    | 适用场景         |
| ----------- | ------------------------------------- | ------------ |
| **按意图分库**   | 保障重叠意图 → 调用「险种责任库」+「重叠逻辑库」；普通问答 → 默认库 | 不同意图检索不同知识范围 |
| **主库 + 辅库** | 主库：百万医疗条款；辅库：学平险/补充医疗/社保              | 保障重叠时同时查多险种  |
| **单库多文档**   | 若暂不拆库，可在同一知识库中上传上述新文档，通过 query 增强召回   | 一期快速落地       |


#### D. 配置扩展（多库支持）

```python
# config/settings.py 扩展

# 默认知识库（沿用现有）
RAGFLOW_KNOWLEDGE_BASE_ID: Optional[str] = "7f6af9a5080c11f1a760fe457711635e"

# 保障重叠度专用：可配置为单个 ID 或 ID 列表（多库组合）
# 单个：RAGFLOW_COVERAGE_KB_IDS = "xxx"
# 多个：RAGFLOW_COVERAGE_KB_IDS = ["id1", "id2"]  或 用逗号分隔字符串 "id1,id2"
RAGFLOW_COVERAGE_KB_IDS: Optional[str] = None  # 空则用默认 RAGFLOW_KNOWLEDGE_BASE_ID
```

调用时：

```python
kb_ids = _parse_coverage_kb_ids()  # 解析为 ["id1", "id2"] 或 [default_id]
call_ragflow(query=enhanced_query, knowledge_base_id=kb_ids, top_k=5)
```

### 5.3 检索提示词 / Query 增强（高效利用 RAGflow）

RAGflow 的 retrieval 接口**没有**单独的「prompt」参数，但可通过 **Query 增强**实现类似「检索提示」的效果，引导召回更贴近意图的内容。

#### A. 思路

在调用 RAGflow 前，根据**意图**对用户问题做改写或追加关键词，使检索更聚焦：

- 原始问题：`学平险浪费吗？`
- 增强后：`学平险 免赔额 门诊 身故 理赔 险种责任 保障重叠 与百万医疗区别`

#### B. 意图 → Query 增强规则（示例）


| 意图                  | 追加关键词                  | 说明           |
| ------------------- | ---------------------- | ------------ |
| `coverage_overlap`  | 免赔额、门诊、住院、身故、险种责任、理赔逻辑 | 强化召回条款与结构类内容 |
| `medical_insurance` | 百万医疗、投保、核保、续保          | 沿用现有改写       |
| `claims`            | 理赔、报案、材料、流程            | 理赔相关         |
| 默认                  | 无追加                    | 仅用问题改写结果     |


#### C. 实现方式

在 `call_ragflow` 调用前增加一层「Query 增强」：

```python
def enhance_query_for_retrieval(query: str, intent: str) -> str:
    """按意图追加检索关键词，提升召回精准度。"""
    if intent == "coverage_overlap":
        suffix = " 免赔额 门诊 住院 身故 险种责任 理赔逻辑"
        return f"{query}{suffix}" if query.strip() else query.strip() + suffix
    # 其他意图可按需扩展
    return query
```

在 pipeline 中：

```python
search_query = rewrite_result["rewritten_query"]
if intent_name == INTENT_COVERAGE_OVERLAP:
    search_query = enhance_query_for_retrieval(search_query, intent_name)
ragflow_result = call_ragflow(
    search_query,
    knowledge_base_id=coverage_kb_ids or None,  # 多库时传列表
    top_k=settings.COVERAGE_OVERLAP_TOP_K,
)
```

#### D. 可选：启用 RAGflow keyword 匹配

RAG 接口支持 `keyword: true`，可同时启用关键词匹配，与向量检索结合，对「免赔额」「门诊」等专业词召回更稳：

```python
payload["keyword"] = True  # 需 ragflow_client 支持透传
```

### 5.4 整体检索链路（多库 + 增强）

```
用户问题（保障重叠意图）
    ↓
问题改写（补全指代）
    ↓
Query 增强（追加：免赔额 门诊 身故 险种责任...）
    ↓
call_ragflow(question=增强后query, dataset_ids=[KB1, KB2, ...], top_k=5, keyword=True)
    ↓
RAGflow 从多库检索，返回合并 chunks
    ↓
compute_coverage_gap + generate_answer
```

## 九、验收标准

### 9.1 功能验收
