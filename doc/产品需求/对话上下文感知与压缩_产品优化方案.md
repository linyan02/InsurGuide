# 对话上下文感知与压缩 —— 产品优化方案

## 一、现状分析

### 1.1 当前对话流程架构

```
用户提问 ──► Redis 取上下文 ──► 意图识别 ──► 问题改写 ──► RAGflow 检索 ──► 答案生成 ──► 存上下文
                  │                    │                    │                  │
                  └────────────────────┴────────────────────┴──────────────────┘
                                              上下文使用点
```

**关键文件**：
- `core/redis_store.py`：存/取对话历史，格式 `[{"query":"...", "answer":"..."}]`，最多 10 轮
- `app/query_rewrite.py`：问题改写，结合上下文补全指代（"那""这个"等）
- `app/answer_engine.py`：答案生成，将**完整 context 直接 json.dumps 进 Prompt**
- `services/rag/langchain_chain.py`：LangChain 模式，同样将完整 context 传入
- `app/llm_short.py`（extract_insurance_slots）：百万医疗险 Slot 提取，使用 context 做要素补全

### 1.2 存在的核心问题

| 问题 | 描述 | 影响 |
|------|------|------|
| **上下文未被有效利用** | 答案生成时虽然传入了 context，但 Prompt 只要求「结合历史上下文」，未明确指导如何关联、何时引用 | 模型可能忽略无关轮次，也可能混淆多轮话题 |
| **无上下文关联性筛选** | 所有历史轮次一视同仁传入，不区分与当前问题的关联度 | 话题跳跃时（如从「蓝医保」切换到「好医保理赔」），旧上下文可能干扰答案 |
| **Token 无控制** | context 用 `json.dumps(context)` 全量传入，无截断、无压缩 | 10 轮 × 每轮 500+ 字 ≈ 5000+ 字 ≈ 1500+ tokens，叠加知识库后易超限、成本高、时延增加 |
| **改写阶段已有截断** | `query_rewrite` 的 LLM 模式仅用最近 5 轮、answer 截断 200 字 | 改写与答案生成对 context 的处理不一致，答案生成仍用全量 context |

### 1.3 典型用户场景

| 场景 | 示例对话 | 期望行为 | 当前可能问题 |
|------|----------|----------|--------------|
| 连续追问 | Q1:蓝医保甲状腺能买吗？ A1:… Q2:那理赔呢？ | Q2 应理解为「蓝医保理赔」，结合 Q1 话题回答 | 改写阶段可补全，答案阶段依赖全量 context |
| 话题切换 | Q1:蓝医保… A1:… Q2:好医保和蓝医保哪个好？ | 应聚焦 Q2，Q1 仅作背景参考 | 可能被旧上下文干扰 |
| 长对话 | 8 轮问答后追问「那续保呢？」 | 需知道前面讨论的是哪款产品 | context 过长，token 压力大 |
| 跨轮对比 | Q1:蓝医保… Q2:好医保… Q3:哪个更适合我？ | 需综合前两轮结论 | 需保留关联轮，压缩无关轮 |

---

## 二、优化目标

1. **上下文感知**：每次回答时，明确考虑与当前问题**关联**的历史轮次
2. **关联性筛选**：只传入与当前问题相关的历史，降低噪声
3. **Token 可控**：对传入 LLM 的上下文实施**压缩与预算**，避免 token 爆炸
4. **保持兼容**：与现有意图识别、问题改写、RAGflow 检索流程兼容，尽量减少侵入式改动

## 三、方案设计

### 3.1 整体架构

```
                    ┌────────────────────────────────────────────────────────────────┐
                    │                    上下文预处理模块（新增）                        │
                    │  get_relevant_context(query, full_context) → compressed_context │
                    └────────────────────────────────────────────────────────────────┘
                                                    │
    用户提问 ──► Redis 取上下文 ──► 意图识别 ──► 问题改写 ──► RAGflow ──► │ 上下文压缩/筛选 │ ──► 答案生成
                  │                    │                    │              │
                  │                    └──────────────────────┴──────────────┘
                  │                          （改写阶段仍用原始 context，保证指代补全）
                  └───────────────────────────────────────────────────────────────────┘
```

**核心改动**：在**答案生成前**增加「上下文压缩/筛选」步骤，产出 `compressed_context` 再传入 `generate_answer`。问题改写阶段继续使用原始 `context`，保证「那理赔呢」等指代能正确补全。

### 3.2 上下文筛选策略（按关联性）

#### 策略 A：最近 N 轮 + 相关性筛选（推荐）

| 规则 | 说明 |
|------|------|
| 最近 1 轮必选 | 上一轮问答与当前问题关联度最高（尤其指代型提问） |
| 语义相似度筛选 | 用当前 query（或改写后 query）与历史各轮 query 计算相似度，取 top-K 相关轮 |
| 实体/主题重叠 | 若当前问句含「蓝医保」「好医保」等实体，优先保留提及同实体的轮次 |
| 兜底 | 若无可用筛选结果，保留最近 2–3 轮 |

**相似度实现**（二选一或组合）：
- **方案 1（轻量）**：关键词/实体重叠（jieba 分词 + 集合交集），无额外依赖
- **方案 2（效果更好）**：文本向量相似度（如 DashScope text-embedding），需调用 Embedding API

#### 策略 B：时间衰减 + 主题聚类（进阶）

- 对每一轮赋予「与当前问题的关联分」= 相似度 × 时间衰减（越近权重越高）
- 按分数排序，取前 K 轮或前 token 预算内的轮次

### 3.3 上下文压缩策略（控制 Token）

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **截断单轮** | 每轮 answer 超过 N 字（如 150）时截断，保留「结论句」或前 N 字 | 实现简单，效果一般 |
| **摘要压缩** | 对较早轮次（如前 5 轮之外的）做 LLM 摘要，如「用户询问了蓝医保承保条件，已解答；后又问了理赔材料，已解答」 | Token 显著下降，需额外 LLM 调用 |
| **分层压缩** | 最近 2 轮完整保留；中间轮次截断；更早的轮次合并为一段摘要 | 平衡质量与 token |
| **Token 预算** | 设定 context 最大 token 数（如 800），按轮从新到旧依次加入，超预算则停止 | 硬约束，保证不超限 |

**推荐组合**：
1. **单轮截断**：每轮 answer 限制 150 字，query 限制 50 字
2. **Token 预算**：context 总 token 不超过 600–800（约 1200–1600 中文字）
3. **分层**：最近 2 轮不截断（或放宽到 200 字），更早的轮次严格截断

### 3.4 配置项设计

```python
# config/settings.py 新增

# ---------- 对话上下文压缩 ----------
# 是否启用上下文压缩（False 则沿用当前全量传入逻辑）
CONTEXT_COMPRESSION_ENABLED: bool = True

# 上下文筛选模式：recent_only | similarity | hybrid
# recent_only：只按时间取最近 N 轮
# similarity：按与当前问题的相似度筛选
# hybrid：最近 1 轮必选 + 相似度筛选其余
CONTEXT_SELECTION_MODE: str = "hybrid"

# 最近必选轮数（hybrid 模式下，最近几轮一定保留）
CONTEXT_RECENT_REQUIRED: int = 1

# 最大保留轮数（筛选后）
CONTEXT_MAX_TURNS: int = 5

# 单轮 answer 最大字数（超出截断，保留前 N 字）
CONTEXT_TURN_ANSWER_MAX_CHARS: int = 150

# 单轮 query 最大字数
CONTEXT_TURN_QUERY_MAX_CHARS: int = 50

# 上下文总 token 预算（约等于 2×中文字数，用于控制总长度）
CONTEXT_MAX_TOKENS: int = 800

# 相似度筛选时，使用的实现：keyword | embedding
# keyword：关键词/实体重叠
# embedding：向量相似度（需配置 embedding API）
CONTEXT_SIMILARITY_METHOD: str = "keyword"
```

### 3.5 模块设计：context_compressor

```
app/context_compressor.py（新增）
├── compress_context(query, context, mode)  # 主入口
├── _select_relevant_turns(query, context)  # 关联性筛选
├── _truncate_turn(turn)                    # 单轮截断
├── _apply_token_budget(turns, max_tokens)  # Token 预算裁剪
└── _keyword_similarity(query, turn_query)  # 关键词相似度
```

**接口约定**：

```python
def compress_context(
    query: str,
    context: List[Dict[str, Any]],
    *,
    mode: Optional[str] = None,  # 覆盖配置
) -> List[Dict[str, Any]]:
    """
    对原始 context 进行筛选与压缩，返回可用于答案生成的上下文。
    - query: 当前用户问题（可用 rewritten_query 以提升筛选准确度）
    - context: Redis 取出的完整历史
    - 返回: 筛选并截断后的 [{"query":"...", "answer":"..."}, ...]
    """
```

### 3.6 与现有流程的集成

| 环节 | 改动 |
|------|------|
| `chat_service.chat_once` | 在调用 `generate_answer` 前，增加 `compress_context(query, context)`，将结果传入 |
| `services/rag/pipeline.run_chat_pipeline` | 同上 |
| `services/rag/langchain_chain.run_chat_with_langchain` | 同上 |
| `app/answer_engine.generate_answer` | 入参 `context` 改为「已压缩的 context」，内部逻辑不变 |
| `app/query_rewrite` | **不变**，继续使用原始 `context` |
| `app/llm_short.extract_insurance_slots` | **不变**，Slot 提取依赖完整上下文，继续用原始 context |

## 四、实现优先级与阶段

### 4.1 一期（最小可行）

- [ ] 实现 `context_compressor.py`
- [ ] 策略：`recent_only` + 单轮截断 + Token 预算
- [ ] 不启用相似度筛选，仅「最近 N 轮 + 每轮截断 + 总 token 上限」
- [ ] 在 `generate_answer` 调用前接入
- [ ] 配置开关 `CONTEXT_COMPRESSION_ENABLED`，默认 True

**预期效果**：Token 从约 1500 降至约 600–800，回答仍能利用最近几轮上下文。

### 4.2 二期（关联性增强）

- [ ] 实现 `keyword` 相似度筛选
- [ ] 启用 `hybrid` 模式：最近 1 轮必选 + 其余按相似度选 top-K
- [ ] 实体词提取（产品名、险种等）辅助筛选

**预期效果**：话题切换时减少无关轮次干扰，长对话下更聚焦当前问题。

### 4.3 三期（可选）

- [ ] `embedding` 相似度（接入 DashScope text-embedding）
- [ ] 较早轮次 LLM 摘要压缩
- [ ] 上下文结构优化：从「平铺 Q&A」改为「概要 + 最近 2 轮完整」等结构化格式

```diff
## 历史上下文：

```

## 六、风险与应对

| 风险 | 应对 |
|------|------|
| 压缩过度导致信息丢失 | 最近 1–2 轮不截断或放宽；提供配置开关可关闭压缩 |
| 相似度筛选漏掉关键轮 | hybrid 模式保证最近轮必选；可调大 `CONTEXT_MAX_TURNS` |
| 改写与答案阶段 context 不一致 | 设计上改写用原始 context，答案用压缩 context，两者职责分离 |
| 新增 LLM 摘要带来时延与成本 | 三期再考虑，一期不做摘要 |

---

## 七、验收标准

1. **Token 控制**：启用压缩后，单次对话 context 部分 token 不超过设定值（如 800）
2. **指代补全**：用户问「那理赔呢」等指代问题，仍能正确理解为上一轮话题的理赔
3. **话题切换**：从 A 产品切换到 B 产品时，回答聚焦 B，不被 A 的旧结论干扰
4. **长对话**：8–10 轮后追问，仍能基于相关轮次给出合理回答
5. **配置生效**：关闭 `CONTEXT_COMPRESSION_ENABLED` 时，行为与当前版本一致
