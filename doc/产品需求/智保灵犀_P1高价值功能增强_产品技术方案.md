# 智保灵犀·P1 高价值功能增强 —— 产品技术方案

> 强化汇报说服力：条款粘贴入口、条款结构化抽取、管理端看板  
> 对应需求来源：`doc/产品需求/智保灵犀_功能增强建议_汇报导向.md` P1 部分  
> 依赖：P0 前端条款解析闭环已完成  
> 文档版本：v1.0 | 生成日期：2026-02-27

---

## 一、需求背景

### 1.1 P1 目标

在 P0 形成「上传 → 解析 → 多轮咨询」闭环的基础上，P1 旨在：

| 功能 | 目标 | 汇报价值 |
|------|------|----------|
| **P1-5 条款粘贴入口** | 显式提供「粘贴条款」可折叠区，降低无 PDF 用户的输入门槛 | PPT 双入口完整，覆盖更多场景 |
| **P1-6 条款结构化抽取** | 条款加载后自动抽取免赔额、等待期、续保条件等关键要素，以卡片展示 | **差异化亮点**：「从解释者到规划师」的过渡 |
| **P1-7 管理端基础看板** | 展示交互量、意图分布、合规拦截、条款解析调用等数据 | **可回溯佐证**：汇报时展示数据落库、可审计 |
| **P1-8 clause_loaded 展示** | 流式响应中展示「基于您上传的条款」 | 用户感知、汇报时一眼看出在用条款 |

**说明**：P1-8 已在 P0 实现中完成，本期仅需在验收时确认。

## 二、P1-5 条款粘贴入口

### 2.1 产品设计

#### 2.1.1 用户旅程

```
1. 用户点击「粘贴条款」展开可折叠区
2. 在文本框中粘贴条款内容（可含问题，如「下面是条款，免赔额怎么算？」）
3. 点击「发送」或折叠区内「解析条款」按钮
4. 系统将长文本作为 clause_text 传入 chat，并展示条款状态栏
5. 用户可继续多轮追问
```

#### 2.1.2 界面设计

```
┌──────────────────────────────────────────────────────────────────┐
│  [条款粘贴区] 可折叠，默认收起                                      │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ ▼ 粘贴条款（无 PDF 时可用）                                   │ │
│  │ ┌─────────────────────────────────────────────────────────┐ │ │
│  │ │ 请在此粘贴保险条款内容，可与问题一起发送...                  │ │ │
│  │ │                                                         │ │ │
│  │ └─────────────────────────────────────────────────────────┘ │ │
│  │                                    [解析条款] [收起]          │ │
│  └─────────────────────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────────────────────┤
│  [条款状态栏] 已有条款时显示（沿用 P0）                              │
│  [文件预览]  ...                                                   │
│  [输入框]  ...                                                     │
└──────────────────────────────────────────────────────────────────┘
```

- **展开/收起**：点击「粘贴条款」标题栏切换，默认收起
- **文本框**：多行，占位符「请在此粘贴保险条款内容，可与问题一起发送」
- **解析条款**：点击时相当于「发送」，将粘贴区内容 + 输入框内容合并处理（见下发逻辑）
- **与输入框关系**：粘贴区可单独使用，也可与输入框配合（输入框填问题，粘贴区填条款）

#### 2.1.3 发送逻辑

| 场景 | 处理 |
|------|------|
| 粘贴区有内容，输入框为空 | `clause_text` = 粘贴区内容；`query` = 若粘贴区含明显问句则提取，否则用「请解读上述条款」 |
| 粘贴区有内容，输入框有内容 | `clause_text` = 粘贴区内容；`query` = 输入框内容 |
| 粘贴区为空 | 沿用 P0 逻辑（输入框文本 >200 字且含条款词时作为 clause_text） |

**问句提取简化规则**：若粘贴区最后一句以「？」结尾且长度 <80 字，视为问题；否则 `query` = 「请解读上述条款」。

### 2.2 技术实现

#### 2.2.1 涉及文件

| 类型 | 文件 | 变更 |
|------|------|------|
| 前端 | `web/static/index.html` | 新增粘贴区 HTML、折叠逻辑、解析按钮、handleSend 分支 |

#### 2.2.2 实现要点

- 粘贴区 `textarea` id：`clause-paste-area`，初始 `rows=4`
- 折叠容器 id：`clause-paste-collapse`，收起时内容区 `hidden`
- 「解析条款」按钮：调用 `handleSend()`，并传入「来自粘贴区」的标记
- `handleSend` 增加分支：若粘贴区有内容，优先用粘贴区作为 `clause_text`，不再依赖「>200 字 + 关键词」推断

## 三、P1-6 条款结构化抽取（简化版）

### 3.1 产品设计

#### 3.1.1 用户旅程

```
1. 用户上传/粘贴条款并完成首轮发送
2. 系统在返回正常解读答案的同时，触发一次「结构化抽取」
3. 抽取结果以卡片形式展示在回答区域下方（或作为首条消息）
4. 卡片字段：免赔额、等待期、续保条件、责任免除要点（能抽则填，无则显示「未在条款中明确」）
```

#### 3.1.2 展示形式

```
┌─────────────────────────────────────────────────────────────────┐
│ 【智保灵犀·条款解读】                                             │
│ ... 正常解读内容 ...                                              │
├─────────────────────────────────────────────────────────────────┤
│ 【关键要素速览】                                                   │
│ ┌──────────┬──────────┬──────────┬────────────────────────────┐ │
│ │ 免赔额   │ 等待期   │ 续保条件 │ 责任免除要点                 │ │
│ ├──────────┼──────────┼──────────┼────────────────────────────┤ │
│ │ 1万元/年 │ 30天     │ 非保证续保│ 既往症、故意行为...          │ │
│ └──────────┴──────────┴──────────┴────────────────────────────┘ │
│ 本解读仅供参考，具体以保险合同条款为准。                           │
└─────────────────────────────────────────────────────────────────┘
```

#### 3.1.3 触发条件

| 条件 | 是否抽取 |
|------|----------|
| 意图 = clause_parse 且存在 clause_ctx 或 clause_text | 是 |
| 首轮（context_count ≤ 1） | 是 |
| 非首轮 | 否（避免重复） |

### 3.2 技术实现

#### 3.2.1 方案选型

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **A. 单次 LLM 调用** | 在 answer_engine 中，条款解析分支多调一次 LLM 做抽取，将 JSON 拼入答案 | 实现简单，无新接口；答案体量增大 |
| **B. 独立接口** | 新增 `POST /api/clause/extract`，前端在条款加载后主动调用 | 前后端解耦；多一次请求 |
| **C. 流式附加** | 在 SSE done 事件中附加 `extracted` 字段，前端单独渲染卡片 | 与现有流式兼容；需扩展事件格式 |

**推荐**：方案 A（首轮条款解析时，在同一轮内完成抽取并写入答案）。前端只需解析 Markdown 中的「关键要素速览」表格并做高亮或卡片样式渲染（可选）。

#### 3.2.2 后端实现要点

| 模块 | 变更 |
|------|------|
| `app/answer_engine.py` | 新增 `extract_clause_structured(clause_text, clause_docs)`，返回 `{deductible, waiting_period, renewal, exclusions}`；在 `generate_answer` 的 clause_parse 分支中，若为首轮且有条款，先调用抽取，将结果格式化为 Markdown 表格拼入 prompt 或答案末尾 |
| `services/rag/pipeline.py` / `pipeline_stream.py` | 传递 `context_count` 或 `is_first_turn` 至 answer_engine，用于判断是否执行抽取 |

#### 3.2.3 抽取 Prompt 示例

```
请从以下保险条款内容中抽取以下关键要素，以 JSON 返回。若某要素未在条款中明确提及，该字段填 null。

要素：
- deductible: 免赔额（如「1万元/年」，简洁表述）
- waiting_period: 等待期（如「30天」）
- renewal: 续保条件（如「非保证续保」「保证续保20年」）
- exclusions: 责任免除要点（2-3 条，每条不超过 30 字）

条款内容：
{clause_content}

仅输出 JSON，不要其他说明。
```

#### 3.2.4 数据流

```
用户发送（clause_parse + 首轮）
  → pipeline 识别意图、取 clause_ctx / clause_text
  → answer_engine：若首轮，先 extract_clause_structured()
  → 抽取结果格式化为 Markdown 表格
  → 拼入 CLAUSE_PARSE_PROMPT 的「已有抽取结果」占位，或拼在答案末尾
  → 返回给前端
```

## 四、P1-7 管理端基础看板（最小可行）

### 4.1 产品设计

#### 4.1.1 目标

- 展示近 7 天核心指标，支撑汇报「可回溯、可审计」
- 不做复杂权限，可用简单密码或与主站共用 JWT 的 admin 角色（若已有）

#### 4.1.2 看板指标

| 指标 | 说明 | 数据来源 |
|------|------|----------|
| 交互总量 | 近 7 天成功对话轮数 | `interaction_logs`，按 `created_at` 过滤 |
| 意图分布 | 各意图的占比（条数） | 需 `interaction_logs` 增加 `intent` 字段 |
| 合规拦截次数 | 近 7 天违规词屏蔽次数 | `compliance_logs`，`violated = true` |
| 条款解析调用次数 | clause_parse 意图的条数 | 同上，依赖 `intent` 字段 |

#### 4.1.3 界面示意

```
┌─────────────────────────────────────────────────────────────────┐
│ 智保灵犀 - 管理看板                                    [退出]    │
├─────────────────────────────────────────────────────────────────┤
│ 数据范围：近 7 天                                                 │
│                                                                  │
│ ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────────┐ │
│ │ 交互总量   │ │ 合规拦截   │ │ 条款解析   │ │ 活跃用户(去重) │ │
│ │   1,234   │ │     12     │ │    89      │ │      56         │ │
│ └────────────┘ └────────────┘ └────────────┘ └────────────────┘ │
│                                                                  │
│ 意图分布                                                         │
│ ┌──────────────────────────────────────────────────────────────┐ │
│ │ 条款检索 35% │ 理赔 22% │ 医疗条款解析 18% │ 保障重叠 12% ... │ │
│ └──────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 技术实现

#### 4.2.1 数据库扩展

| 表 | 变更 | 说明 |
|------|------|------|
| `interaction_logs` | 新增 `intent` 列（VARCHAR(64)，可选） | 用于意图分布、条款解析次数统计 |

**迁移**：新增列可为 NULL，不影响旧数据；新写入时由 pipeline 传入 intent。

#### 4.2.2 后端变更

| 模块 | 变更 |
|------|------|
| `models/chat_log.py` | `InteractionLog` 增加 `intent = Column(String(64), nullable=True)` |
| `services/rag/pipeline.py` | `save_interaction_log` 增加 `intent` 参数并写入 |
| `app/chat_service.py` | 同上（若仍使用 chat_service 路径） |
| `routers/` | 新增 `admin.py` 或 `routers/admin.py`：`GET /api/admin/dashboard` 返回统计 JSON |
| `api/main.py` | 注册 admin 路由 |

#### 4.2.3 统计接口

**GET /api/admin/dashboard**

- Query：`days=7`（默认 7）
- 鉴权：需登录，可增加 `is_admin` 或固定 admin 密码（最小可行）
- 响应示例：

```json
{
  "code": 200,
  "data": {
    "total_interactions": 1234,
    "compliance_violations": 12,
    "clause_parse_count": 89,
    "active_users": 56,
    "intent_distribution": [
      {"intent": "retrieval", "intent_cn": "条款检索", "count": 432},
      {"intent": "clause_parse", "intent_cn": "医疗条款解析", "count": 89}
    ]
  }
}
```

#### 4.2.4 前端

| 形式 | 说明 |
|------|------|
| **独立页面** | `web/static/admin.html` 或 `/admin` 路由，调用上述接口并渲染 |
| **Gradio Tab** | 在 Gradio 中增加「管理看板」Tab，调用接口展示（可选） |

最小可行：单页 `admin.html`，表格/卡片展示各指标，无需复杂图表。

## 五、P1-8 clause_loaded 展示（已实现）

P0 实现中已在 SSE `done` 事件解析 `clause_loaded`，并在回答旁展示「📄 基于您上传的条款」。

**验收**：条款解析场景下，流式完成时若后端返回 `clause_loaded: true`，前端应显示该标识。若后端暂未返回，需在 `pipeline_stream` 的 clause_parse 分支 `done` 事件中补充该字段。

## 六、验收标准

### 6.1 P1-5 条款粘贴入口

| 序号 | 验收项 | 预期 |
|------|--------|------|
| F1 | 粘贴区可折叠 | 点击展开/收起，默认收起 |
| F2 | 粘贴条款 + 输入问题发送 | 答案基于粘贴的条款 |
| F3 | 仅粘贴条款发送 | 能正确解析并展示条款状态 |
| F4 | 与文件上传不冲突 | 粘贴区与文件选择可独立使用，逻辑清晰 |

### 6.2 P1-6 条款结构化抽取

| 序号 | 验收项 | 预期 |
|------|--------|------|
| F1 | 首轮条款解析 | 回答中包含「关键要素速览」表格（免赔额、等待期等） |
| F2 | 非首轮 | 不重复抽取 |
| F3 | 要素缺失 | 未在条款中明确的要素显示「未明确」或留空 |

### 6.3 P1-7 管理端看板

| 序号 | 验收项 | 预期 |
|------|--------|------|
| F1 | 交互总量 | 近 7 天 interaction_logs 条数 |
| F2 | 合规拦截 | 近 7 天 compliance_logs 中 violated=true 条数 |
| F3 | 条款解析次数 | intent=clause_parse 的 interaction_logs 条数 |
| F4 | 意图分布 | 各 intent 的条数及占比 |

### 6.4 P1-8 clause_loaded

| 序号 | 验收项 | 预期 |
|------|--------|------|
| F1 | 条款解析回答 | 显示「📄 基于您上传的条款」 |
