# 智保灵犀·P0 前端条款解析闭环 —— 产品技术方案

> 补齐 PC Web 前端与条款解析后端的对接，形成「上传 → 解析 → 多轮咨询」完整闭环  
> 对应需求来源：`doc/产品需求/智保灵犀_功能增强建议_汇报导向.md` P0 部分  
> 文档版本：v1.0 | 生成日期：2026-02-27

---

## 一、需求背景

### 1.1 问题陈述

后端已实现医疗条款解析能力（`/api/clause/upload`、`clause_context`、多库检索），但 PC Web 前端存在以下断点：

| 断点 | 现状 | 影响 |
|------|------|------|
| 文件未上传 | 用户选择 PDF 后点击发送，仅将文本发送至 chat，文件从未调用 clause/upload | 无法实现「上传 PDF → 问免赔额」核心演示 |
| 无 session_id | chat 请求不传 session_id | clause_ctx 无法按会话绑定，多轮咨询时条款上下文丢失 |
| 无条款状态展示 | 无「已加载 xxx.pdf」、无清除按钮 | 用户与汇报时无法感知当前是否有条款上下文 |
| 无 clause_text 传递 | 用户粘贴长条款时，未作为 clause_text 传入 chat | 粘贴条款场景不可用 |

### 1.2 目标

补齐前端与后端条款解析的对接，形成**可演示的完整闭环**：

1. 用户上传 PDF/Word → 文件先入库 → 再发起咨询 → 多轮追问基于该条款回答
2. 用户粘贴长条款 → 与问题一起发送 → 基于该条款回答
3. 条款状态可视、可清除

## 二、产品设计

### 2.1 用户旅程

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 方式 A：文件上传                                                          │
├─────────────────────────────────────────────────────────────────────────┤
│ 1. 用户点击 📎 选择 PDF/Word/TXT 文件                                     │
│ 2. 文件预览显示：「已选：xxx.pdf」                                         │
│ 3. 用户输入问题（可选）或直接点击发送                                        │
│ 4. 系统：先调用 clause/upload 上传并解析 → 成功后再调用 chat/stream       │
│ 5. 输入框上方展示：「已加载：xxx.pdf」 [清除]                               │
│ 6. 用户可继续多轮提问，无需重复上传                                         │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 方式 B：粘贴条款                                                          │
├─────────────────────────────────────────────────────────────────────────┤
│ 1. 用户在输入框粘贴长文本（>200 字且含条款关键词）                           │
│ 2. 可与问题一起发送，或先发条款再发问题                                      │
│ 3. 系统：将长文本作为 clause_text 传入 chat，后端存入 clause_ctx            │
│ 4. 输入框上方展示：「已加载：条款文本」 [清除]                               │
│ 5. 用户可继续多轮提问                                                     │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 界面设计

#### 2.2.1 输入区布局（修改后）

```
┌──────────────────────────────────────────────────────────────────┐
│  [条款状态栏] 仅在有条款时显示                                      │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ 📄 已加载：某某医疗险条款.pdf          [清除]                   │ │
│  └─────────────────────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────────────────────┤
│  [文件预览] 仅在有文件选择时显示                                     │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ DOC  xxx.pdf                                    [×]          │ │
│  └─────────────────────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────────────────────┤
│  [输入框]                                                          │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ 在此输入保险咨询问题...                                        │ │
```

#### 2.2.2 条款状态栏

| 状态 | 展示内容 | 操作 |
|------|----------|------|
| 无条款 | 不显示 | - |
| 已加载（文件） | `📄 已加载：{file_name}` + [清除] 按钮 | 点击清除 → 调用 /api/clause/clear，隐藏状态栏 |
| 已加载（粘贴） | `📄 已加载：条款文本` + [清除] 按钮 | 同上 |

#### 2.2.3 发送逻辑（核心变更）

| 场景 | 发送前动作 | chat 请求体 |
|------|------------|-------------|
| 仅有文本 | 无 | `{ user_id, query, session_id, model_plan }` |
| 有文件 | 先 `POST /api/clause/upload`，成功后再 chat | 同上 + `session_id` 必传 |
| 有长条款文本 | 无 | `{ user_id, query, session_id, clause_text, model_plan }` |
| 有文件 + 文本 | 先 upload，再 chat | `{ user_id, query, session_id, model_plan }`（条款已在 upload 中入库，无需 clause_text） |

### 2.3 会话与 session_id

| 操作 | session_id 行为 |
|------|-----------------|
| 打开页面 / 登录 | 生成新 session_id：`sess_${Date.now()}_${random}` |
| 点击「新对话」 | 重置为新的 session_id，清除条款状态栏 |
| 发送消息 | 始终带当前 session_id |
| 恢复历史对话 | 沿用当前 session_id（不恢复当时条款；P2 可扩展） |

### 2.4 条款粘贴识别规则（P0-4）

当用户**仅发送文本**（无文件）且满足以下条件时，将文本作为 `clause_text` 传入：

- 文本长度 > 200 字
- 且包含以下任一关键词：`条款`、`合同`、`保险责任`、`免赔`、`等待期`、`续保`、`责任免除`

满足时：`clause_text = 用户输入的完整文本`，`query = 用户输入` 或 若用户明确分离了问题则用问题部分（本期简化：整段作为 clause_text，query 用「请解读上述条款」或用户输入）。

**简化实现**：若文本 > 200 字，统一作为 clause_text 传入，query 取前 100 字作为摘要或固定为「请根据上述条款回答我的问题」（实际以用户输入为准，若用户输入是纯条款则后端会识别意图）。

**更简实现**：当 `text.length > 200` 时，`clause_text = text`，`query = text`（后端会从 clause_text 提取用于检索，query 用于展示）。产品上可约定：用户粘贴条款时可带一句问题，如「下面是条款，免赔额怎么算」，则拆分为 clause_text = 条款部分，query = 「免赔额怎么算」。P0 可简化为：整段传入 clause_text，query 为用户输入原样。

## 三、技术实现

### 3.1 涉及文件

| 类型 | 文件 | 变更说明 |
|------|------|----------|
| 前端 | `web/static/index.html` | 上传对接、session_id、条款状态栏、clause_text、chat 请求体扩展 |
| 后端 | 无 | 接口已就绪，无需修改 |

### 3.2 前端实现要点

#### 3.2.1 session_id 管理

```javascript
// 生成 session_id
function generateSessionId() {
  return 'sess_' + Date.now() + '_' + Math.random().toString(36).slice(2, 10);
}

let currentSessionId = generateSessionId();

// 新对话时调用
function onNewChat() {
  currentSessionId = generateSessionId();
  clearClauseState();  // 隐藏条款状态栏、清除本地状态
  // 可选：调用 /api/clause/clear 清除服务端
}
```

#### 3.2.2 条款上传流程（有文件时）

```javascript
async function uploadClauseFile(file, sessionId) {
  const formData = new FormData();
  formData.append('file', file);
  formData.append('session_id', sessionId);

  const r = await fetch(API_BASE + '/api/clause/upload', {
    method: 'POST',
    headers: { 'Authorization': 'Bearer ' + getToken() },
    body: formData,
  });

  if (!r.ok) {
    const err = await r.json().catch(() => ({}));
    throw new Error(err.detail || '条款上传失败');
  }
  return await r.json();
}
```

**接口约定**：

- `POST /api/clause/upload`
- Content-Type: `multipart/form-data`
- Body: `file`（文件）, `session_id`（可选，表单字段）
- Header: `Authorization: Bearer <token>`
- 响应 200: `{ message, dataset_id, file_name }`
- 响应 4xx/5xx: `{ detail: "错误信息" }`

**注意**：当前 `routers/clause.py` 的 `session_id` 为可选参数，FastAPI 对 multipart 通常通过表单字段接收。若接口未显式声明 `Form()`，需确认是否支持 `session_id` 作为 form field。若不支持，可考虑 query 参数：`/api/clause/upload?session_id=xxx`。实际实现时以服务端接受方式为准。

#### 3.2.3 条款清除

```javascript
async function clearClause() {
  const r = await fetch(API_BASE + '/api/clause/clear', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': 'Bearer ' + getToken(),
    },
    body: JSON.stringify({ session_id: currentSessionId }),
  });
  if (r.ok) {
    hideClauseStatusBar();
    clauseFileName = null;
  }
}
```

#### 3.2.4 chat/stream 请求体扩展

```javascript
// 原：{ user_id, query, model_plan }
// 现：{ user_id, query, session_id, clause_text?, model_plan }

const body = {
  user_id: currentUser.username,
  query: queryToSend,
  session_id: currentSessionId,
  model_plan: modelPlan || 'standard',
};

// 当用户粘贴了长条款（无文件）时，增加 clause_text
if (!hasFile && text.length > 200 && looksLikeClause(text)) {
  body.clause_text = text;
}

const r = await fetch(API_BASE + '/api/chat/stream', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(body),
});
```

`looksLikeClause` 简化实现：检查是否包含 `条款|合同|保险责任|免赔|等待期|续保|责任免除` 等关键词。

#### 3.2.5 handleSend 主流程（伪代码）

```javascript
async function handleSend() {
  const text = (input.value || '').trim();
  const hasFile = !fileBox.classList.contains('hidden');
  const file = document.getElementById('f-upload')?.files?.[0];

  if (!text && !hasFile) return;

  // 1. 若有文件，先上传条款
  if (hasFile && file) {
    try {
      showUploadingState();  // 可选：显示「正在解析条款…」
      const uploadRes = await uploadClauseFile(file, currentSessionId);
      showClauseStatusBar(uploadRes.file_name);  // 显示「已加载：xxx.pdf」
    } catch (e) {
      showError('条款上传失败：' + e.message);
      return;
    } finally {
      cancelFile();
    }
  }

  // 2. 确定 query
  const queryToSend = text || (hasFile ? '请根据我上传的条款做条款解析与保障建议' : '');
  if (!queryToSend) return;

  // 3. 构建 chat 请求体
  const body = {
    user_id: currentUser.username,
    query: queryToSend,
    session_id: currentSessionId,
    model_plan: document.getElementById('model-select')?.value || 'standard',
  };
  if (!hasFile && text.length > 200 && looksLikeClause(text)) {
    body.clause_text = text;
  }

  // 4. 调用 chat/stream，处理 SSE...
}
```

#### 3.2.6 后端 session_id 传参方式校验

`routers/clause.py` 的 `clause_upload` 中 `session_id: Optional[str] = None` 默认为 query 参数。若需从 form 传，需改为：

```python
session_id: Optional[str] = Form(None)
```

建议在实现前确认：上传接口是否接受 `session_id`。若仅支持 query，则前端使用：

```
POST /api/clause/upload?session_id=sess_xxx
```

### 3.3 接口一览

| 接口 | 方法 | 鉴权 | 说明 |
|------|------|------|------|
| `/api/clause/upload` | POST | Bearer | 上传条款文件，Form: file, session_id? |
| `/api/clause/context` | GET | Bearer | 获取当前会话条款上下文，Query: session_id? |
| `/api/clause/clear` | POST | Bearer | 清除条款上下文，Body: { session_id? } |
| `/api/chat/stream` | POST | 否（user_id 区分） | 流式对话，Body 增加 session_id、clause_text? |

### 3.4 后端适配说明（如需）

当前 `routers/clause.py` 的 `clause_upload` 签名：

```python
session_id: Optional[str] = None
```

在 FastAPI 中，对于 `UploadFile` 的端点，其他参数默认来自 query。若前端通过 `FormData` 传 `session_id`，需在路由中显式声明为 `Form(None)`，否则不会从 body 解析。建议：

- 方案 A：前端用 query 传 `session_id`，`/api/clause/upload?session_id=xxx`
- 方案 B：后端改为 `session_id: Optional[str] = Form(None)`，前端在 formData 中 append

产品方案中以方案 A 为默认，实现简单且与现有参数风格一致。

## 四、验收标准

### 4.1 功能验收

| 序号 | 验收项 | 操作步骤 | 预期 |
|------|--------|----------|------|
| F1 | 文件上传入库 | 选择 PDF → 输入「免赔额怎么算」→ 发送 | 先上传成功，再返回基于条款的答案 |
| F2 | 多轮咨询 | 上传后连续问「等待期多久」「续保条件」 | 均基于同一份条款回答，无需重新上传 |
| F3 | 条款状态展示 | 上传成功后 | 输入框上方显示「已加载：xxx.pdf」 |
| F4 | 清除条款 | 点击 [清除] | 状态栏消失，下一问不再基于该条款 |
| F5 | session_id 传递 | 任意发送 | 请求体包含 session_id |
| F6 | 粘贴条款 | 粘贴 >200 字条款 → 问「免赔额怎么算」→ 发送 | 答案基于粘贴的条款 |
| F7 | 新对话 | 点击「新对话」 | session_id 重置，条款状态栏消失 |

### 4.2 非功能验收

| 序号 | 验收项 | 预期 |
|------|--------|------|
| NF1 | 上传失败提示 | 格式不支持、超时等给出明确错误信息 |
| NF2 | 上传中状态 | 大文件时可显示「正在解析条款…」避免用户重复点击 |
| NF3 | 与现有功能兼容 | 不破坏登录、历史记录、流式输出等原有能力 |
