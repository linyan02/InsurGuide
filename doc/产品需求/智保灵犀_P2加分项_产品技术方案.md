## 二、P2-10 Gradio 条款解析入口

### 2.1 产品设计

#### 2.1.1 目标

在 Gradio「对话服务」Tab 的「自建增强 RAG」区域，增加与 PC Web 一致的条款解析能力：上传条款、展示状态、清除条款，支持多轮条款咨询。

#### 2.1.2 用户旅程

```
1. 用户在 Gradio 选择「自建增强 RAG」模式
2. 输入 user_id 后，可点击「上传条款」选择 PDF/Word/TXT
3. 上传成功后显示「已加载：xxx.pdf」及「清除」按钮
4. 输入问题发送，请求带 session_id，后端基于 clause_ctx 回答
5. 可继续多轮追问，或点击「清除」后切换其他条款
```

#### 2.1.3 界面示意

```
┌─────────────────────────────────────────────────────────────────────┐
│ 智保灵犀：基于 RAGflow 知识库的多轮对话                              │
├─────────────────────────────────────────────────────────────────────┤
│ 用户 ID    [___________________]                                    │
│                                                                      │
│ ┌─ 条款状态 ─────────────────────────────────────────────────────┐ │
│ │ 📄 已加载：百万医疗险条款.pdf          [清除]                     │ │
│ └──────────────────────────────────────────────────────────────────┘ │
│ （或无条款时： [📎 上传条款] 按钮）                                  │
│                                                                      │
│ 提问       [_______________________________________________________] │
│                                                                      │
│                                    [发送]                            │
├─────────────────────────────────────────────────────────────────────┤
│ 答案                                                                  │
│ ...                                                                   │
└─────────────────────────────────────────────────────────────────────┘
```

#### 2.1.4 交互逻辑

| 场景 | 处理 |
|------|------|
| 无条款 | 显示「上传条款」按钮；发送时 chat 不传 clause_text、session_id（沿用默认 session_id="default"） |
| 有条款 | 显示「已加载：xxx.pdf」+ 清除；发送时 chat 带 session_id（与 clause_ctx 绑定） |
| 清除 | 调用 POST /api/clause/clear，更新 UI 为无条款状态 |

### 2.2 技术实现

#### 2.2.1 涉及文件

| 类型 | 文件 | 变更 |
|------|------|------|
| 配置 | `config/constants.py` | 新增 GRADIO_ROUTE_CLAUSE_UPLOAD、CLAUSE_CONTEXT、CLAUSE_CLEAR |
| 配置 | `gradio_ui/config.py` | 新增 API_CLAUSE_UPLOAD、API_CLAUSE_CONTEXT、API_CLAUSE_CLEAR |
| Gradio | `gradio_ui/pages/chat.py` | 增加上传按钮、条款状态栏、清除按钮；chat 请求带 session_id；需鉴权 |

#### 2.2.2 Gradio 鉴权约束

当前 Gradio 对话接口 `/api/chat` 不强制鉴权；但 `/api/clause/upload`、`/api/clause/clear`、`/api/clause/context` 需 `Authorization: Bearer <token>`。

**方案 A（推荐）**：Gradio 条款功能仅在「已登录」时可用。若 Gradio 侧未做登录态持久化，则：
- 在对话 Tab 增加「登录」入口（或复用已有认证 Tab 的登录结果）
- 登录成功后保存 token 到 Gradio 的 Session 或全局变量
- 调用 clause 相关 API 时带 token

**方案 B（最小可行）**：暂不鉴权，后端 `clause` 接口改为可选鉴权（dev 环境），Gradio 用 user_id 作为标识。生产环境再收紧。

**推荐**：方案 A，与 PC Web 一致，需在 Gradio 中建立 token 获取与传递机制。

#### 2.2.3 实现要点

- **session_id**：每个 Gradio 会话（或每次「新对话」）生成 `sess_<timestamp>_<random>`，与 PC Web 一致
- **上传**：使用 `gr.File` 或 `gr.UploadButton`，选择文件后调用 `POST /api/clause/upload`（multipart/form-data，需 token + session_id）
- **状态栏**：上传成功后展示 `file_name`；调用 `GET /api/clause/context?session_id=xxx` 可拉取当前状态（可选，用于刷新）
- **清除**：点击清除时调用 `POST /api/clause/clear`，body 含 `session_id`
- **chat 请求**：`_chat_rag` 中增加 `session_id` 参数，且需调用流式接口 `POST /api/chat/stream` 以支持 clause_loaded 等字段（可选：当前非流式也可先支持 session_id）

#### 2.2.4 API 路由常量补充

```
GRADIO_ROUTE_CLAUSE_UPLOAD = "/api/clause/upload"
GRADIO_ROUTE_CLAUSE_CONTEXT = "/api/clause/context"
GRADIO_ROUTE_CLAUSE_CLEAR = "/api/clause/clear"
GRADIO_ROUTE_CHAT_STREAM = "/api/chat/stream"  # 若 Gradio 改用流式
```

## 三、P2-11 对话历史恢复时 clause 关联

### 3.1 产品设计

#### 3.1.1 目标

用户点击侧栏「最近记录」中某条历史并选择「继续此对话」时，不仅恢复对话上下文（问+答），还恢复该条记录对应的条款上下文（clause_ctx），使多轮咨询在「有条款」状态下延续。

#### 3.1.2 用户旅程

```
1. 用户 A 上传条款后进行了 3 轮条款咨询
2. 用户新建对话或关闭页面，clause_ctx 可能已过期（Redis TTL）或 session_id 已切换
3. 用户点击「最近记录」中第 2 轮那条，选择「继续此对话」
4. 系统恢复：该轮的问答上下文 + 当时的条款上下文（若可恢复）
5. 用户继续提问「那责任免除呢」，基于原条款回答
```

#### 3.1.3 技术难点

- **clause_ctx 存于 Redis**：key 为 `clause_ctx:{user_id}:{session_id}`，有过期时间
- **历史记录存于 MySQL**：`interaction_logs` 仅有 `user_id, query, answer`，无 `session_id`、无 clause 关联
- **恢复时**：需要知道「那条记录当时用的 session_id」以及「当时的 clause_ctx 快照」，才能恢复

### 3.2 技术实现

#### 3.2.1 数据模型扩展

| 表 | 变更 | 说明 |
|------|------|------|
| `interaction_logs` | 新增 `session_id` 列（VARCHAR(64)，可选） | 记录该轮对话对应的会话 ID |
| `interaction_logs` | 新增 `clause_snapshot` 列（TEXT/JSON，可选） | 该轮若基于条款，存储 clause_ctx 的可恢复快照。source=text 时 text_full 可截断至 16KB 以防列过长 |

**clause_snapshot 结构**（与 Redis clause_ctx 一致，便于直接写回）：

```json
{
  "source": "upload",
  "dataset_id": "xxx",
  "file_name": "百万医疗.pdf",
  "text_preview": null,
  "text_full": null,
  "created_at": 1738012345
}
```

- **source=upload**：恢复时需 dataset_id（RAGflow 知识库仍存在则可复用）、file_name 用于展示
- **source=text**：恢复时需 text_full（粘贴的完整条款）。若 text_full 超长（如 >64KB），可只存 text_preview，恢复时提示「条款文本已过期，请重新粘贴」

#### 3.2.2 写入逻辑

| 模块 | 变更 |
|------|------|
| `models/chat_log.py` | `InteractionLog` 增加 `session_id`、`clause_snapshot` |
| `services/rag/pipeline.py` / `pipeline_stream.py` | `save_interaction_log` 增加 `session_id`、`clause_snapshot` 参数 |
| 调用处 | 在 clause_parse 分支或「有 clause_ctx」时，将 clause_ctx 序列化为 JSON 传入 |

**写入时机**：仅当该轮对话实际使用了 clause_ctx（即 clause_loaded=true 或进入 clause_parse 分支）时写入 clause_snapshot，避免无关记录膨胀。

#### 3.2.3 恢复逻辑

| 模块 | 变更 |
|------|------|
| `routers/chat.py` | `restore_context`：根据 `log_id` 查出 `session_id`、`clause_snapshot` |
| `app/clause_context.py` | 新增 `restore_clause_context(user_id, session_id, snapshot)`：将 snapshot 写回 Redis |
| 前端 | 恢复后可选：刷新条款状态栏（若有 file_name 则显示「已恢复：xxx.pdf」） |

**restore 流程**：

```
1. 查出 InteractionLog(row) by log_id + user_id
2. clear_conversation_context(user_id)
3. save_conversation_context(user_id, row.query, row.answer)
4. 若 row.clause_snapshot 非空：
   a. 生成或沿用 session_id（优先用 row.session_id，否则新建）
   b. restore_clause_context(user_id, session_id, row.clause_snapshot)
5. 返回 { session_id, clause_restored: true/false, file_name? }
```

#### 3.2.4 限制与降级

| 场景 | 处理 |
|------|------|
| clause_snapshot 为 source=upload，但 RAGflow 侧 dataset 已被删 | 无法恢复，返回 clause_restored=false，仅恢复对话文本 |
| clause_snapshot 为 source=text，text_full 未存（历史数据） | 无法恢复完整条款，仅恢复对话文本 |
| 旧数据无 session_id、clause_snapshot | 仅恢复对话上下文，与当前行为一致 |

## 四、P2-12 智能推荐问题

### 4.1 产品设计

#### 4.1.1 目标

条款加载后（上传或粘贴），在输入区附近展示 3～5 个「推荐问题」快捷按钮，用户点击即作为 query 发送，降低首次使用门槛，汇报演示时流程更顺畅。

#### 4.1.2 用户旅程

```
1. 用户上传/粘贴条款，条款状态栏显示「已加载」
2. 推荐问题区域展示：「免赔额怎么算」「等待期多久」「续保条件是什么」「责任免除有哪些」
3. 用户点击「免赔额怎么算」
4. 系统将该句作为 query 发送，正常走 clause_parse 流程
```

#### 4.1.3 界面示意

```
┌─────────────────────────────────────────────────────────────────────┐
│ 📄 已加载：百万医疗险条款.pdf                    [清除]             │
├─────────────────────────────────────────────────────────────────────┤
│ 推荐问题（点击即问）                                                 │
│ [免赔额怎么算] [等待期多久] [续保条件是什么] [责任免除有哪些]        │
├─────────────────────────────────────────────────────────────────────┤
│ [输入框] ...                                               [发送]    │
└─────────────────────────────────────────────────────────────────────┘
```

- **展示时机**：仅当条款状态栏可见（已加载条款）时展示
- **隐藏时机**：清除条款后隐藏
- **位置**：条款状态栏与输入框之间，或输入框上方

#### 4.1.4 推荐问题列表

| 序号 | 问题文本 |
|------|----------|
| 1 | 免赔额怎么算 |
| 2 | 等待期多久 |
| 3 | 续保条件是什么 |
| 4 | 责任免除有哪些 |
| 5 | 请解读上述条款 |

可配置：前端写死常量，或由后端 `GET /api/clause/suggested-questions` 返回（便于后续扩展）。

### 4.2 技术实现

#### 4.2.1 方案选型

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **A. 前端写死** | 常量数组，条款加载时展示 | 实现简单，无新接口 |
| **B. 后端接口** | `GET /api/clause/suggested-questions` 返回列表 | 可配置、可扩展（如按产品类型推荐） |

**推荐**：方案 A 最小可行；若后续需按条款类型或产品定制，再演进为 B。

#### 4.2.2 涉及文件

| 类型 | 文件 | 变更 |
|------|------|------|
| 前端 | `web/static/index.html` | 新增推荐问题区域 HTML；条款加载时显示、清除时隐藏；点击时调用 `handleSend` 并传入问题文本 |

#### 4.2.3 实现要点

- **容器**：`<div id="suggested-questions" class="hidden ...">`，与条款状态栏联动
- **展示条件**：`showClauseStatusBar` 时同时 `showSuggestedQuestions`；`hideClauseStatusBar` 时 `hideSuggestedQuestions`
- **点击逻辑**：点击按钮时，将按钮文本填入输入框并触发发送；或直接 `handleSend` 且 `queryToSend = 按钮文本`、不修改输入框
- **Gradio**：若 P2-10 已完成，在 Gradio 条款状态栏下方同样增加推荐问题按钮，点击即调用 `_chat_rag` 并传入该问题

## 五、验收标准

### 5.1 P2-10 Gradio 条款解析入口

| 序号 | 验收项 | 预期 |
|------|--------|------|
| F1 | 上传条款 | Gradio 可选择文件并上传，成功后在状态栏显示文件名 |
| F2 | 条款咨询 | 上传后发送问题，答案基于条款（clause_parse） |
| F3 | 清除条款 | 点击清除后状态栏消失，再发问不走条款解析 |
| F4 | session_id | chat 请求携带 session_id，与 clause_ctx 正确绑定 |

### 5.2 P2-11 对话历史恢复时 clause 关联

| 序号 | 验收项 | 预期 |
|------|--------|------|
| F1 | 有条款历史恢复 | 恢复某条「当时有条款」的历史，clause_ctx 一并恢复，可继续基于条款追问 |
| F2 | 无条款历史恢复 | 恢复某条「当时无条款」的历史，仅恢复问答上下文，行为与现有一致 |
| F3 | 旧数据兼容 | 无 clause_snapshot 的历史记录，恢复时仅恢复问答，不报错 |

### 5.3 P2-12 智能推荐问题

| 序号 | 验收项 | 预期 |
|------|--------|------|
| F1 | 展示时机 | 条款加载后显示推荐问题区域 |
| F2 | 隐藏时机 | 清除条款后隐藏 |
| F3 | 点击即问 | 点击推荐问题后，该问题作为 query 发送并得到回答 |
