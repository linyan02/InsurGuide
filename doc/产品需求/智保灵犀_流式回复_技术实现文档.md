# 智保灵犀·流式回复 —— 技术实现文档

> 目标：将 `web/static/index.html` 中的问题回复从「整段一次性返回」改为「流式逐字/逐句输出」，提升用户感知体验。

---

## 一、现状分析

### 1.1 当前调用链路

```
用户点击发送 → index.html handleSend()
    → fetch POST /api/chat { user_id, query, model_plan }
    → routers/chat.py chat()
    → run_chat_pipeline() 或 run_chat_with_langchain()
        → 意图识别 → 问题改写 → RAGflow 检索 → generate_answer() → 合规校验 → 保存上下文
    → 返回完整 JSON { code, data: { answer, source, intent_cn, ... } }
    → 前端一次性渲染 contentDiv.innerHTML = renderAnswer(rawAnswer)
```

### 1.2 问题点

| 环节 | 现状 | 影响 |
|------|------|------|
| 前端 | `await fetch()` 等待完整响应，再 `data = await r.json()` | 用户长时间看到「思考中…」，无即时反馈 |
| 后端 | `generate_answer()` 内部 `_call_dashscope()` 为同步非流式调用 | 必须等 LLM 生成完整文本后才返回 |
| LLM 调用 | `httpx.post(..., json={...})` 无 `stream: true` | DashScope/OpenAI 返回完整 result，无 chunk |

### 1.3 关键文件

| 文件 | 职责 |
|------|------|
| `web/static/index.html` | 发送请求、展示「思考中…」、一次性渲染答案 |
| `routers/chat.py` | `/api/chat` 接口，返回 JSON |
| `services/rag/pipeline.py` | 主流程：意图→改写→RAGflow→答案→合规 |
| `app/answer_engine.py` | `generate_answer()`、`_call_dashscope()` 非流式 LLM 调用 |

---

## 二、方案选型

### 2.1 流式技术路线

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **SSE (Server-Sent Events)** | 使用 `text/event-stream`，前端 `EventSource` 或 `fetch + ReadableStream` | 单向、轻量、易实现，适合「服务器推文本」 |
| **WebSocket** | 双向通道 | 过重，当前只需服务端→客户端推送 |
| **长轮询** | 定时请求 | 延迟高、资源浪费 |

**推荐**：SSE + `StreamingResponse`，与 OpenAI/DashScope 兼容模式流式返回格式一致。

### 2.2 接口设计

**新增流式接口** `POST /api/chat/stream`，与现有 `/api/chat` 并行，避免破坏现有调用方。

- **请求体**：与 `/api/chat` 相同（`user_id`、`query`、`model_plan` 等）
- **响应**：`Content-Type: text/event-stream`
- **事件格式**：
  - `data: {"type":"chunk","content":"..."}` —— 文本片段
  - `data: {"type":"done","answer":"...","source":[...],"intent_cn":"..."}` —— 结束，含完整答案与元数据

### 2.3 流程拆分

流式仅作用于 **LLM 生成答案** 阶段，前置流程（意图、改写、RAGflow）仍为同步：

```
1. 意图识别、问题改写、RAGflow 检索（与现有一致，非流式）
2. 若为引导态（GUIDING）：直接返回 JSON 式事件，前端按非流式处理
3. 若为完整回答：调用 LLM 流式 API，逐 chunk 推送到前端
4. 流结束后，发送元数据（source、intent_cn 等），再保存上下文与日志
```

## 三、技术实现方案

### 3.1 后端：流式 LLM 调用

#### 3.1.1 DashScope 兼容模式流式 API

项目使用 `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`（OpenAI 兼容），支持 `stream: true`：

```python
# 请求
resp = httpx.post(
    url,
    headers={"Authorization": f"Bearer {key}"},
    json={
        "model": model_name,
        "messages": [{"role": "user", "content": prompt}],
        "stream": True,  # 启用流式
    },
    timeout=60,
)
# 响应：SSE 格式，每行 data: {...}
# 每个 chunk: {"choices":[{"delta":{"content":"..."}}]}
# 结束: data: [DONE]
```

```python
def _call_dashscope_stream(prompt: str, model: Optional[str] = None):
    """流式调用 DashScope，yield 每个 content 片段。"""
    model_name = (model or "qwen-turbo").strip() or "qwen-turbo"
    import httpx
    with httpx.stream(
        "POST",
        "https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions",
        headers={"Authorization": f"Bearer {settings.DASHSCOPE_API_KEY}"},
        json={
            "model": model_name,
            "messages": [{"role": "user", "content": prompt}],
            "stream": True,
            "max_tokens": 1024,
        },
        timeout=60.0,
    ) as resp:
        for line in resp.iter_lines():
            if not line or line.strip() != line:
                continue
            if line.startswith("data: "):
                data = line[6:]
                if data.strip() == "[DONE]":
                    break
                try:
                    import json
                    obj = json.loads(data)
                    choices = obj.get("choices") or []
                    if choices and isinstance(choices[0], dict):
                        delta = choices[0].get("delta") or {}
                        content = delta.get("content")
                        if content:
                            yield content
                except Exception:
                    pass
```

### 3.3 后端：流式路由

#### 3.3.1 `routers/chat.py` 新增

```python
from fastapi.responses import StreamingResponse

@router.post("/chat/stream")
async def chat_stream(
    body: ChatRequest = Body(...),
    db: Session = Depends(get_db),
):
    """流式对话接口，返回 SSE。"""
    user_id = (body.user_id or "").strip()
    query = (body.query or "").strip()
    if not user_id or not query:
        raise HTTPException(status_code=400, detail="user_id 和 query 不能为空")

    async def event_generator():
        try:
            from services.rag.pipeline_stream import run_chat_pipeline_stream
            async for event in run_chat_pipeline_stream(
                db, user_id, query,
                intent_mode=body.intent_mode,
                rewrite_mode=body.rewrite_mode,
                model_plan=body.model_plan,
            ):
                yield f"data: {json.dumps(event, ensure_ascii=False)}\n\n"
        except Exception as e:
            yield f"data: {json.dumps({'type':'error','message':str(e)}, ensure_ascii=False)}\n\n"

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
        },
    )
```

### 3.4 前端：流式消费

#### 3.4.1 `index.html` 修改要点

1. **新增 `handleSendStream`**：使用 `fetch` + `ReadableStream` 消费 SSE，不再 `await r.json()`。
2. **占位符渲染**：先插入「思考中…」占位 div，收到 `chunk` 时逐步追加文本，收到 `done` 时补全元数据并替换为 Markdown 渲染。
3. **兼容逻辑**：可通过配置或 query 参数选择流式/非流式，或默认走流式。

#### 3.4.2 核心 JS 示例

```javascript
async function handleSendStream() {
    // ... 前置校验、插入用户消息、占位符 ...
    
    const r = await fetch(API_BASE + '/api/chat/stream', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
            user_id: currentUser.username,
            query: queryToSend,
            model_plan: document.getElementById('model-select').value || 'standard'
        })
    });
    
    if (!r.ok) {
        contentDiv.innerHTML = '请求失败：' + r.status;
        return;
    }
    
    const reader = r.body.getReader();
    const decoder = new TextDecoder();
    let buffer = '';
    let fullText = '';
    
    while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        buffer += decoder.decode(value, { stream: true });
        const lines = buffer.split('\n\n');
        buffer = lines.pop() || '';
        
        for (const line of lines) {
            if (!line.startsWith('data: ')) continue;
            const data = line.slice(6);
            try {
                const event = JSON.parse(data);
                if (event.type === 'chunk') {
                    fullText += event.content || '';
                    contentDiv.textContent = fullText;  // 或边收边渲染
                } else if (event.type === 'done') {
                    fullText = event.answer || fullText;
                    const html = renderAnswer(fullText);
                    if (event.source && event.source.length)
                        html += '<div class="mt-4...">来源：' + event.source.join('、') + '</div>';
                    contentDiv.classList.add('ai-answer-content');
                    contentDiv.innerHTML = html;
                } else if (event.type === 'error') {
                    contentDiv.textContent = '错误：' + (event.message || '未知');
                }
            } catch (_) {}
        }
    }
    
    scrollToBottom();
}
```

#### 3.4.3 Markdown 流式渲染策略

- **方案 A**：流式过程中用 `textContent` 显示纯文本，`done` 后再用 `marked.parse()` 转 Markdown。
- **方案 B**：每收到 chunk 都对 `fullText` 做 `marked.parse()`，实时渲染（可能产生频闪，不推荐）。
- **推荐**：方案 A，体验更稳定。

## 五、验收与测试

### 5.1 功能验收

| 项 | 预期 |
|------|------|
| 流式展示 | 发送问题后，答案区域逐字/逐句出现，而非长时间「思考中…」后一次性显示 |
| 元数据 | 流结束后，来源、意图等正常展示 |
| 错误处理 | RAGflow/LLM 异常时，前端展示错误信息 |
| 引导态 | 要素未补全时，直接展示追问（可为非流式单次事件） |

### 5.2 性能与兼容

- 首字延迟应明显低于「等完整答案」的体感时间。
- 现有 `POST /api/chat` 保持可用，不破坏 Gradio、测试脚本等调用方。
- 若需支持「带登录的流式」，在 `chat_stream` 中可增加 `Depends(get_current_active_user)`，并从 token 解析 `user_id`。
