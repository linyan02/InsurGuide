# 智保灵犀增强 RAG 调用系统 - 详细技术实现方案

本文档对应 `doc/智保灵犀增强RAG调用系统产品技术方案.pdf`，描述当前代码实现与模块划分，便于开发与维护。

---

## 一、方案目标与架构回顾

- **目标**：RAGflow 独立构建知识库，增强 RAG 服务负责用户交互与推理；知识库管理与业务交互解耦。
- **能力**：多轮对话、意图识别与答案生成、RAGflow 检索、合规校验、上下文与日志存储。

整体数据流：

1. 用户通过 Web/小程序/API 发起提问；
2. 增强 RAG 服务解析用户标识、加载多轮对话上下文（Redis）；
3. **意图识别**：对用户问题做意图分类（检索/咨询/理赔/核保/产品/问候/其他）；
4. **问题改写**：结合历史上下文将指代、省略句改写成可独立检索的完整问题；
5. 用**改写后的问题**调用 RAGflow 检索 API，传入知识库 ID；
6. RAGflow 返回知识库匹配片段与来源；
7. 增强 RAG 结合**原始问题**与上下文、保险领域 Prompt、轻量 LLM 生成答案；
8. 合规校验（违规词屏蔽）→ 写交互/合规日志（MySQL）→ 更新上下文（Redis）→ 返回答案、溯源来源、意图与改写结果。

## 三、技术栈与配置

| 模块           | 选型              | 说明 |
|----------------|-------------------|------|
| Web 框架       | FastAPI (Python 3.9+) | 异步、自动 OpenAPI 文档 |
| 多轮对话存储   | Redis 7.0         | Key: `context:{user_id}`，TTL 30 分钟，最多 10 轮 |
| 交互/合规日志  | MySQL             | 表：interaction_logs、compliance_logs |
| 知识库检索     | RAGflow HTTP API  | 文档解析、向量化、检索由 RAGflow 完成 |
| 答案生成       | 通义千问 / OpenAI 兼容 API，或本地 Qwen-1.8B | 由 `LLM_MODE` 与 API Key 决定 |
| 合规校验       | 内置违规词库       | 环境变量 `VIOLATION_WORDS`，逗号分隔 |

配置集中在 `config.py` 与 `.env`，包括：Redis、RAGflow URL/Key/知识库 ID、LLM 模式与 API Key、违规词列表等。

## 四、模块与代码映射

### 4.1 多轮对话上下文管理

- **文档要求**：Redis 缓存，Key 为用户 ID，Value 为对话历史，过期 30 分钟，最多 10 轮。
- **实现**：`app/context_store.py`
  - `get_conversation_context(user_id)`：取当前用户历史列表 `[{"query","answer"}, ...]`；
  - `save_conversation_context(user_id, query, answer)`：追加一轮并截断到最近 N 轮、刷新 TTL；
  - `clear_conversation_context(user_id)`：清除该用户上下文（供 `/api/chat/clear` 使用）。
- **配置**：`REDIS_*`、`REDIS_CONTEXT_TTL_MINUTES`、`REDIS_MAX_HISTORY_TURNS`。
- **依赖**：Redis 需先安装并配置，否则上下文功能不可用，接口仍可响应。

### 4.2 意图识别（多种模式）

- **目标**：对用户问题做保险场景意图分类，支持按业务场景选择调用方式。
- **实现**：`app/intent.py`
  - **模式**：`rule`（规则）、`llm`（LLM）、`llm_vector`（LLM + 向量库意图规则）、`bert`（自训练 BERT API）。
  - **规则模式** `recognize_rule(query)`：基于关键词匹配，无外部 API，低延迟。
  - **LLM 模式** `recognize_llm(query)`：大模型直接输出意图标签，需配置 API Key。
  - **LLM+向量库模式** `recognize_llm_vector(query)`：先从向量库（`INTENT_RULES_COLLECTION`）检索与问题相似的意图规则，再将规则作为上下文交给 LLM 判断；规则可热更新，适合可配置场景。
  - **BERT 模式** `recognize_bert(query)`：调用自训练 BERT 意图分类 API（部署在独立服务器）。**兜底策略**：若请求超时（`BERT_INTENT_TIMEOUT`，默认 2 秒）或网络/解析失败，自动切换为兜底模式（`BERT_INTENT_FALLBACK_MODE`，默认 `rule`），返回中 `method` 为 `bert_fallback`，并带 `fallback_reason`（如 `timeout`、`http_500`）。
  - **统一入口** `recognize(query, mode=None)`：mode 为空时使用配置 `INTENT_MODE`；支持请求体传入 `intent_mode` 覆盖。
- **配置**：`INTENT_MODE`（rule | llm | llm_vector | bert）、`INTENT_RULES_COLLECTION`、`INTENT_VECTOR_TOP_K`；BERT 相关：`BERT_INTENT_API_URL`、`BERT_INTENT_TIMEOUT`、`BERT_INTENT_FALLBACK_MODE`、`BERT_INTENT_REQUEST_QUERY_KEY`（可选，请求体字段名，默认 `text`）。
- **BERT API 约定**（可适配）：请求 `POST` JSON 默认 `{"text": "用户问题"}`；响应需含意图与置信度，支持字段名：`intent`/`label`/`class`、`confidence`/`score`/`probability`。标签会归一化为 `ALL_INTENTS` 之一，否则归为 `other`。
- **向量库规则写入**：`POST /api/intent-rules/add`，Body 为 `[{ "content": "意图：claims。关键词：理赔、赔付...", "rule_id": "可选" }]`。

### 4.3 用户问题改写（三种模式）

- **目标**：结合多轮对话上下文将指代、省略改写成可独立检索的完整问题，支持按业务场景选择调用方式。
- **实现**：`app/query_rewrite.py`
  - **模式**：`rule`、`llm`、`llm_vector`。
  - **规则模式** `rewrite_rule(query, context)`：问句过短或含指代/省略词时，用最近一轮主题补全。
  - **LLM 模式** `rewrite_llm(query, context)`：大模型根据历史直接改写，需配置 API Key。
  - **LLM+向量库模式** `rewrite_llm_vector(query, context)`：从向量库（`REWRITE_RULES_COLLECTION`）检索与「当前问句+上下文」相似的改写示例，作为 few-shot 交给 LLM；示例可热更新。
  - **统一入口** `rewrite(query, context, mode=None)`：mode 为空时使用配置 `REWRITE_MODE`；支持请求体传入 `rewrite_mode` 覆盖。
- **配置**：`REWRITE_MODE`、`REWRITE_RULES_COLLECTION`、`REWRITE_VECTOR_TOP_K`。
- **向量库示例写入**：`POST /api/rewrite-rules/add`，Body 为 `[{ "content": "原问：那理赔呢。上下文：重疾险。改写：重疾险理赔条件。", "rule_id": "可选" }]`。
- **流程**：检索阶段使用 `rewritten_query` 调用 RAGflow；展示与日志仍使用用户原始 `query`。

### 4.4 RAGflow 调用模块

- **文档要求**：调用 RAGflow 检索 API，传提问与知识库 ID，返回匹配文本与来源。
- **实现**：`app/ragflow_client.py`
  - `call_ragflow(query, knowledge_base_id=None, top_k=None)`：
    - 使用 `RAGFLOW_API_URL`、`RAGFLOW_API_KEY`、`RAGFLOW_KNOWLEDGE_BASE_ID`、`RAGFLOW_TOP_K`；
    - 支持 Body 内 `api_key` 或 Header `Authorization: Bearer <key>`；
    - 返回统一结构：`{ "documents": [...], "metadatas": [{ "source", ... }] }` 或 `{ "error": "..." }`；
  - 兼容文档式「knowledge/search」与部分 OpenAI 兼容返回（从 `choices[].message.reference` 抽取 chunks）。
- **说明**：若实际 RAGflow 版本接口路径/格式不同，只需调整 `RAGFLOW_API_URL` 或在此模块内做适配。

### 4.5 合规校验模块

- **文档要求**：内置违规词库，实时检测答案并替换为「[违规表述已屏蔽]」，记录合规日志。
- **实现**：`app/compliance.py`
  - `get_violation_words()`：从 `settings.get_violation_words_list()` 读取（来自 `VIOLATION_WORDS`）；
  - `check_and_mask(answer)`：返回 `(处理后答案, 是否发生违规)`。
- **实现**：合规日志写入在 `app/chat_service.py` 中，在检测到违规时调用 `save_compliance_log(...)`。

### 4.6 答案生成与优化

- **文档要求**：基于 RAGflow 检索结果 + 历史上下文 + 保险领域 Prompt 模板，用轻量 LLM 生成答案。
- **实现**：`app/answer_engine.py`
  - `INSURANCE_PROMPT_TEMPLATE`：与产品文档一致的保险领域说明（引用知识库、禁止编造、结尾合规声明）；
  - `build_knowledge_content(ragflow_result)`：从 RAGflow 返回的 documents/metadatas 拼成「【来源：xxx】内容」文本；
  - `generate_answer(query, ragflow_result, context, do_compliance=True)`：拼 Prompt → 调 LLM → 可选执行合规校验；
  - `call_light_llm(prompt)`：
    - `LLM_MODE=api`：优先 `DASHSCOPE_API_KEY`（通义千问），其次 `OPENAI_API_KEY`（OpenAI 或兼容）；
    - `LLM_MODE=local`：本地 Qwen-1.8B（CPU），需安装 transformers/torch 并下载模型；
  - 无可用 LLM 时返回兜底文案（提示配置 API 或本地模型）。

### 4.7 交互日志与合规日志

- **文档要求**：记录用户提问、答案、合规检测结果，支持审计。
- **实现**：
  - **模型**：`models/chat_log.py`
    - `InteractionLog`：user_id、query、answer、source_count、created_at；
    - `ComplianceLog`：user_id、query、answer_snapshot、violated、remark、created_at。
  - **写入**：`app/chat_service.py`
    - `save_interaction_log(db, user_id, query, answer, source_count)`：每轮对话后写入；
    - `save_compliance_log(db, user_id, query, answer_snapshot, violated, remark)`：仅在发生违规时写入（如 remark="违规表述屏蔽"）。
  - 表在应用启动时通过 `Base.metadata.create_all(bind=engine)` 自动创建（需先 `import models.chat_log`）。

### 4.8 核心接口与编排

- **文档要求**：多轮对话接口，接收 user_id、query，返回 answer、溯源来源、当前轮数等。
- **实现**：`routers/chat.py`、`app/chat_service.py`
  - **POST /api/chat**
    - Body：`{ "user_id": "xxx", "query": "重疾险等待期多久" }`；
    - 流程：取上下文 → **意图识别** → **问题改写** → `call_ragflow(rewritten_query)` → `generate_answer(原始 query, ...)` → `check_and_mask(...)` → 写合规日志（若违规）→ `save_conversation_context` → `save_interaction_log` → 返回 `{ code, message, data: { answer, context_count, source, violated, intent, intent_cn, rewritten_query, rewrite_changed } }`；
  - **POST /api/chat/clear**
    - Body：`{ "user_id": "xxx" }`，清除该用户 Redis 上下文。
- **主应用**：`main.py` 中注册 `chat.router`，并 `import models.chat_log` 以创建日志表。

## 五、配置项汇总

| 环境变量 / 配置 | 说明 | 默认/示例 |
|-----------------|------|-----------|
| REDIS_HOST / PORT / DB / PASSWORD | Redis 连接 | localhost, 6379, 0 |
| REDIS_CONTEXT_TTL_MINUTES | 上下文过期时间（分钟） | 30 |
| REDIS_MAX_HISTORY_TURNS | 最多保留对话轮数 | 10 |
| RAGFLOW_API_URL | RAGflow 检索 API 地址 | http://localhost:9380/v1/search |
| RAGFLOW_API_KEY | RAGflow 鉴权密钥 | - |
| RAGFLOW_KNOWLEDGE_BASE_ID | 知识库/数据集 ID | - |
| RAGFLOW_TOP_K | 检索返回条数 | 3 |
| LLM_MODE | api / local | api |
| DASHSCOPE_API_KEY | 通义千问 API Key | - |
| OPENAI_API_KEY | OpenAI 或兼容 API Key | - |
| VIOLATION_WORDS | 违规词，逗号分隔 | 保证赔付,100%理赔,... |
| INTENT_MODE | 意图识别模式 rule/llm/llm_vector/bert | rule |
| BERT_INTENT_API_URL | 自训练 BERT 意图 API 地址 | - |
| BERT_INTENT_TIMEOUT | BERT 请求超时（秒），超时走兜底 | 2.0 |
| BERT_INTENT_FALLBACK_MODE | BERT 超时/失败时兜底模式 rule/llm | rule |
| REWRITE_MODE | 问题改写模式 rule/llm/llm_vector | rule |
| INTENT_RULES_COLLECTION | 意图规则向量集合名 | intent_rules |
| REWRITE_RULES_COLLECTION | 改写示例向量集合名 | rewrite_rules |
| INTENT_VECTOR_TOP_K / REWRITE_VECTOR_TOP_K | 向量检索条数 | 5 / 3 |

**意图与改写模式选择建议**：
- **rule**：无 API、低延迟，适合离线或对成本敏感场景；意图/改写逻辑固定。
- **llm**：效果更好、泛化能力强，需配置大模型 API，适合对准确率要求高的场景。
- **llm_vector**：在 LLM 基础上引入向量库中的规则/示例，规则可热更新（通过 `/api/intent-rules/add`、`/api/rewrite-rules/add` 写入），适合需要按业务定制、可配置规则或 few-shot 的场景。
- **bert**：自训练 BERT 部署在独立服务器，通过 HTTP API 调用；设短超时（如 2s），超时或失败时自动兜底为 rule/llm，保证可用性。请求体可传 `intent_mode`、`rewrite_mode` 覆盖全局配置，实现按请求灵活选择。

## 七、扩展与后续迭代

- **意图与改写**：已支持 rule / llm / llm_vector 三种模式，可按业务选择；规则与示例可通过向量库 API 热更新。
- **高频问题缓存**：可将「问题指纹 → 答案」缓存到 Redis，TTL 1 小时，减少 RAGflow 与 LLM 调用。
- **OCR / GPU**：文档后续方向；当前答案生成支持 API 与本地 CPU 两种方式，后续可接入 GPU 或 OCR 能力时在 `answer_engine` 或上游解析链路扩展。
