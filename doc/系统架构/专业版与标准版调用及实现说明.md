# 智保灵犀「专业版」与「标准版」调用及代码修改说明

本文档说明前端 index 页中「智保灵犀 - 专业版」与「智保灵犀 - 标准版」的**当前状态**、**拟议调用关系**及**需要修改的代码位置与示例**，便于按需实现两套链路或仅做模型区分。

---

## 一、当前状态

| 项目 | 说明 |
|------|------|
| **前端** | `web/static/index.html` 头部有下拉框「智保灵犀 - 专业版」「智保灵犀 - 标准版」，但发送消息时**未将选中项传给后端**，仅传 `user_id`、`query`。 |
| **后端接口** | `POST /api/chat` 请求体仅有 `user_id`、`query`，以及可选的 `intent_mode`、`rewrite_mode`，**无 model/版本 相关参数**。 |
| **实际调用链** | 由配置 `USE_LANGCHAIN_RAG` 决定走 LangChain 链或原有 pipeline，两路最终均使用**同一套 LLM**：通义千问 **qwen-turbo**（在 `app/answer_engine._call_dashscope` 中写死）。 |

结论：**专业版与标准版目前无任何区别，也未对应两条不同调用链路。**

---

## 二、拟议设计：专业版 / 标准版 的区分方式

### 2.1 约定

- **标准版**：使用 `qwen-turbo`，成本与延迟较低，适合常规咨询。
- **专业版**：使用 `qwen-plus`（或后续更强模型），效果更好，适合复杂条款与深度分析。

可选扩展（按需）：

- 专业版：增大 RAGflow 检索条数 `top_k`，或使用不同知识库 ID。
- 标准版：保持现有 `RAGFLOW_TOP_K` 与默认知识库。

### 2.2 调用关系（实现后）

```
用户选择「专业版」或「标准版」
    ↓
前端 POST /api/chat 携带 model_plan: "pro" | "standard"（可选，缺省为 "standard"）
    ↓
routers/chat.py 解析 body.model_plan，传入 pipeline / langchain_chain
    ↓
┌─ USE_LANGCHAIN_RAG = False ─────────────────────────────────────────┐
│  run_chat_pipeline(..., model_plan=...)                               │
│    → generate_answer(..., model=根据 model_plan 解析出的模型名)       │
│    → call_light_llm(prompt, model=...) → _call_dashscope(prompt, model) │
└──────────────────────────────────────────────────────────────────────┘
┌─ USE_LANGCHAIN_RAG = True ───────────────────────────────────────────┐
│  run_chat_with_langchain(..., model_plan=...)                        │
│    → DashScopeLLM(model_name=根据 model_plan 解析出的模型名)          │
│    → 内部调用 _call_dashscope(prompt, model=...) 或自建请求时传 model │
└──────────────────────────────────────────────────────────────────────┘
```

## 三、代码修改说明

### 3.1 前端：`web/static/index.html`

**目标**：把用户选中的「专业版/标准版」以 `model_plan` 形式随 `POST /api/chat` 传给后端。

- **位置**：发送请求处，即 `fetch(API_BASE + '/api/chat', { ... body: JSON.stringify({ ... }) })`。
- **修改要点**：
  1. 为下拉框 `<select id="model-select">` 的每个 `<option>` 设置 `value`，例如：  
     - 专业版：`<option value="pro">智保灵犀 - 专业版</option>`  
     - 标准版：`<option value="standard">智保灵犀 - 标准版</option>`
  2. 在组装请求体时读取当前选中值，例如：  
     `var modelPlan = document.getElementById('model-select').value || 'standard';`  
     `body: JSON.stringify({ user_id: currentUser.username, query: queryToSend, model_plan: modelPlan })`

**请求体示例**（修改后）：

```json
{
  "user_id": "zhangsan",
  "query": "重疾险等待期一般多长？",
  "model_plan": "pro"
}
```

---

### 3.2 后端：请求体与路由

**文件**：`routers/chat.py`

- **ChatRequest**：增加可选字段 `model_plan`。
  - 类型：`Optional[str] = None`
  - 取值约定：`"pro"`（专业版）、`"standard"`（标准版），空或未传时按标准版处理。
- **chat 视图**：从 `body.model_plan` 取值，并传入下游：
  - 若 `settings.USE_LANGCHAIN_RAG` 为 True：调用 `run_chat_with_langchain(..., model_plan=body.model_plan)`。
  - 若为 False：调用 `run_chat_pipeline(..., model_plan=body.model_plan)`。

**示例（仅展示新增/改动部分）**：

```python
class ChatRequest(BaseModel):
    user_id: str
    query: str
    intent_mode: Optional[str] = None
    rewrite_mode: Optional[str] = None
    model_plan: Optional[str] = None   # "pro" | "standard"，空则按 standard

# 在 chat() 内：
result = run_chat_with_langchain(
    db, user_id, query,
    intent_mode=body.intent_mode,
    rewrite_mode=body.rewrite_mode,
    model_plan=body.model_plan,
)
# 或
result = run_chat_pipeline(
    db, user_id, query,
    intent_mode=body.intent_mode,
    rewrite_mode=body.rewrite_mode,
    model_plan=body.model_plan,
)
```

---

### 3.3 后端：模型名解析（建议集中在一处）

建议在配置或工具模块中集中做「model_plan → DashScope 模型名」的映射，避免多处硬编码。

**可选位置**：`config/settings.py` 增加方法，或新建 `app/model_plan.py`。

**示例**：

```python
# 例如在 config/settings.py 或 app/model_plan.py 中
def get_dashscope_model_for_plan(model_plan: Optional[str]) -> str:
    if model_plan == "pro":
        return "qwen-plus"   # 或 "qwen-max" 等
    return "qwen-turbo"     # standard 或空
```

后续所有需要「按版本选模型」的地方都调用该函数，便于以后改模型名或增加新版本。

---

### 3.4 后端：Pipeline 链路（非 LangChain）

**文件**：`services/rag/pipeline.py`

- **run_chat_pipeline**：增加关键字参数 `model_plan: Optional[str] = None`。
- 在调用 `generate_answer` 前，根据 `model_plan` 得到模型名：  
  `model = get_dashscope_model_for_plan(model_plan)`  
  然后调用：  
  `generate_answer(query, ragflow_result, context, do_compliance=False, model=model)`  
  （若当前 `generate_answer` 无 `model` 参数，需在其签名与内部实现中增加，见下。）

**文件**：`app/answer_engine.py`

- **generate_answer**：增加可选参数 `model: Optional[str] = None`，默认可为 `None` 表示使用默认模型（如 qwen-turbo）。内部在调用 `call_light_llm` 时传入该参数。
- **call_light_llm**：增加可选参数 `model: Optional[str] = None`，并传给 `_call_api_llm(prompt, model=model)`。
- **_call_api_llm**：增加可选参数 `model: Optional[str] = None`，在调用 DashScope 时使用该模型名；若未传则用默认（如 qwen-turbo）。
- **_call_dashscope**：  
  - 签名改为 `_call_dashscope(prompt: str, model: Optional[str] = None) -> str`。  
  - 请求体中的 `"model"` 使用 `model or "qwen-turbo"`，不再写死 `"qwen-turbo"`。

这样，Pipeline 链路从「用户选择专业版/标准版」到「最终调用 DashScope 的模型」就打通了。

---

### 3.5 后端：LangChain 链路

**文件**：`services/rag/langchain_chain.py`

- **run_chat_with_langchain**：增加关键字参数 `model_plan: Optional[str] = None`。
- 在构建 LLM 时根据 `model_plan` 解析模型名：  
  `model_name = get_dashscope_model_for_plan(model_plan)`  
  然后使用：  
  `llm = DashScopeLLM(model_name=model_name)`  
  确保 Chain 使用的 LLM 是按版本选中的模型。

**文件**：`services/rag/langchain_dashscope_llm.py`

- **DashScopeLLM**：已存在 `model_name` 字段，当前 `_call` 内部仍调用 `_call_dashscope(prompt)`，未传模型名。
- **修改**：  
  - 将 `_call_dashscope` 改为支持传入模型名（上节已在 `app/answer_engine` 中改为 `_call_dashscope(prompt, model=...)`）。  
  - 在 `DashScopeLLM._call` 中调用：  
    `text = _call_dashscope(prompt, model=self.model_name)`  
  这样 LangChain 链也会按前端传入的 `model_plan` 使用对应模型。

---

### 3.6 可选：按版本区分检索参数

若希望专业版使用更多检索结果或不同知识库，可在以下位置根据 `model_plan` 分支：

- **RAGflow 检索**：  
  - 在 `run_chat_pipeline` 或 `run_chat_with_langchain` 中，若 `model_plan == "pro"`，可将 `top_k` 调大（或使用 `settings` 中新增的 `RAGFLOW_TOP_K_PRO`），再传入 `call_ragflow` 或 RAGflowRetriever。  
  - 若使用不同知识库，可在调用 RAGflow 时传入不同的 knowledge_base_id（需 RAGflow 客户端支持）。

无需区分时，检索部分可保持不变。

## 四、修改清单汇总

| 序号 | 文件 | 修改内容 |
|------|------|----------|
| 1 | `web/static/index.html` | 为 model-select 的 option 设置 value；请求体中增加 `model_plan`。 |
| 2 | `routers/chat.py` | ChatRequest 增加 `model_plan`；chat 视图将 `model_plan` 传入 run_chat_*。 |
| 3 | 配置或工具 | 新增 `get_dashscope_model_for_plan(model_plan)`，映射 pro → qwen-plus，standard/空 → qwen-turbo。 |
| 4 | `app/answer_engine.py` | `_call_dashscope(prompt, model=None)` 使用传入的 model；`generate_answer` / `call_light_llm` / `_call_api_llm` 增加 model 参数并向下传递。 |
| 5 | `services/rag/pipeline.py` | `run_chat_pipeline` 增加 `model_plan`，解析为模型名后传入 `generate_answer(..., model=...)`。 |
| 6 | `services/rag/langchain_chain.py` | `run_chat_with_langchain` 增加 `model_plan`，按 model_plan 实例化 `DashScopeLLM(model_name=...)`。 |
| 7 | `services/rag/langchain_dashscope_llm.py` | `_call` 中调用 `_call_dashscope(prompt, model=self.model_name)`。 |
| 8 | （可选）RAGflow 调用处 | 根据 `model_plan` 使用不同 top_k 或知识库 ID。 |

## 五、测试建议

1. **前端**：选择「标准版」发送一条消息，在浏览器开发者工具中确认请求体包含 `model_plan: "standard"`；再选「专业版」确认为 `model_plan: "pro"`。
2. **后端**：对两种 model_plan 各发若干请求，在日志或 DashScope 侧确认专业版请求使用 qwen-plus、标准版使用 qwen-turbo（若已接监控/日志）。
3. **兼容性**：不传 `model_plan` 或传空时，行为应与当前一致（全部走 qwen-turbo），保证旧客户端不受影响。

按上述顺序修改即可实现「专业版 / 标准版」的调用区分与对应代码落地；若仅需文档阶段，可仅实现到路由层接收 `model_plan` 并传入 pipeline/chain，下游暂仍使用统一模型，后续再按本文档补齐模型与检索差异。
