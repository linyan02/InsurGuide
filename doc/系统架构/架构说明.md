## 一、总体架构

本系统采用**分层架构**，自上而下分为：前端展示层、API 接口层、增强 RAG 服务层、核心基础设施层、配置层，便于扩展（如小程序端复用 API）、维护与测试。

```
                    ┌─────────────────────────────────────────────────────────┐
                    │                    前端展示层 (web/)                       │
                    │              PC Web（可扩展小程序独立前端）                  │
                    └───────────────────────────┬─────────────────────────────┘
                                                │ HTTP / 静态资源
                    ┌───────────────────────────▼─────────────────────────────┐
                    │                    API 接口层 (api/)                      │
                    │   认证 / 对话 / 向量 / ES / 规则管理  供 Web 与小程序调用   │
                    └───────────────────────────┬─────────────────────────────┘
                                                │ 调用
                    ┌───────────────────────────▼─────────────────────────────┐
                    │                增强 RAG 服务层 (services/)                │
                    │  意图识别 → 问题改写 → 召回 → 融合 → 精排 → 答案生成与合规   │
                    └───────────────────────────┬─────────────────────────────┘
                                                │ 依赖
                    ┌───────────────────────────▼─────────────────────────────┐
                    │               核心基础设施层 (core/)                      │
                    │   数据库 / Redis / 向量库 / ES / 认证                     │
                    └───────────────────────────┬─────────────────────────────┘
                                                │ 配置
                    ┌───────────────────────────▼─────────────────────────────┐
                    │                   配置层 (config/)                         │
                    │               settings / 常量 / 环境变量                   │
                    └─────────────────────────────────────────────────────────┘
```

## 二、目录与职责

| 目录 | 职责 | 说明 |
|------|------|------|
| **doc/** | 项目与架构文档 | 产品技术方案、详细技术实现、架构说明、项目说明 |

| **models/** | 数据模型 | ORM 实体（用户、日志等），与 core 数据库绑定 |
| **utils/** | 通用工具 | 与业务无关的公共函数 |

## 三、增强 RAG 流水线（services/rag）

增强 RAG 是业务核心，采用**管道式**设计，便于单步扩展与替换：

1. **意图识别**（intent）：识别用户意图（检索/咨询/理赔/核保等），支持 rule / llm / llm_vector / bert。
2. **问题改写**（rewrite）：结合多轮上下文将指代、省略改写成可独立检索的完整问题。
3. **召回层**（recall）：多路召回（如 RAGflow、本地向量库），返回候选片段。
4. **融合层**（fusion）：合并多路召回结果、去重、统一格式。
5. **精排层**（rerank）：对融合后的候选做打分/排序（可选），截断 top_k。
6. **答案生成与合规**：基于最终检索结果 + 上下文生成答案，合规校验后返回。

流水线入口为 `services.rag.pipeline`，API 层仅调用流水线与日志/上下文，不直接依赖 RAGflow 或向量库实现细节。

## 四、扩展性要点

- **API 层**：同一套接口供 PC Web 与小程序使用；通过 CORS、鉴权区分环境。
- **召回/精排**：新增一路召回或更换精排模型时，仅改 `services.rag` 内对应模块与配置。
- **意图/改写**：新增模式（如新模型 API）时，在 `services.rag.intent` / `services.rag.rewrite` 中扩展策略，由配置或请求参数选择。
- **配置**：所有环境相关与开关集中在 `config/`，便于多环境部署与特性开关。

## 五、运行与部署

- **API 服务**：`python main.py` 或 `uvicorn api.main:app --host 0.0.0.0 --port 8000`，对外提供接口与文档。
- **PC Web**：静态资源由 API 服务挂载（如 `/static`、`/`）或独立 Nginx 部署，前端请求相对路径或配置的 API 基地址。
- **中间件**：MySQL、Redis、RAGflow、可选 ES、向量库等，见 `README_MIDDLEWARE.md`。

详见 **doc/项目说明.md** 与根目录 **README.md**。
